# Agent 运行失败与容错恢复｜面试题

> 依据：Sitor「吃透 AI Agent 开发」课程笔记、Agent Loop / Tool Pipeline / Gateway 架构、Claude Code / OpenClaw 六大支柱整理  
> 适用目标：AI 应用开发 / Agent Runtime / Tool Calling / Context Engineering / Harness 工程面试

---

## 0. 这组题真正考什么？

这组题不绑定某个业务场景，也不是考某个工具命令，而是考：

> 一个生产级 Agent 在长任务、多轮工具调用、网络波动、上下文膨胀、模型不可靠的情况下，如何保持可控、可恢复、可观测。

面试官可能会用这些问题切入：

- 工具调用失败怎么办？
- 模型一直重复调用同一个工具怎么办？
- 连接中断后怎么恢复任务？
- 上下文爆了怎么办？
- 工具返回太长怎么办？
- 模型 API 流式中断怎么办？
- Memory 和当前事实冲突怎么办？
- 多 Agent 并发执行出问题怎么办？

回答时不要只说“try-catch / 重试 / 截断”，要站在 **Agent Runtime / Harness Engineering** 角度讲：

- Loop 要有退出条件和保险丝。
- Tool 要有校验、权限、超时、幂等和结果过滤。
- Context 要做分层压缩和当前工作集恢复。
- State 要持久化，连接中断后能恢复。
- Observability 要记录工具调用、错误、权限和状态。

---

## 1. 总体回答框架

遇到 Agent 失败问题，可以按五层拆：

1. **模型层**：模型可能幻觉、选错工具、参数错误、输出截断、流式中断。
2. **Agent Loop 层**：可能无限循环、重复调用、步数过多、token 预算烧穿。
3. **Tool Pipeline 层**：工具可能参数非法、业务校验失败、权限不足、超时、返回过大、外部 API 报错。
4. **Context Engineering 层**：上下文可能过长、旧状态污染、关键信息被裁剪、工具结果噪声太大。
5. **Harness / Runtime 层**：连接中断、任务状态丢失、多 Agent 竞态、审计不足、恢复策略缺失。

一句话总结：

> Agent 失败不是单点 bug，而是模型、工具、上下文、状态和运行时边界共同作用的问题；生产级 Agent 要靠系统工程兜底，而不是靠模型自觉。

## 2. 面试题与参考答案

### Q1：如果 Agent 运行失败，你会从哪些层面排查原因？

**参考回答：**

我会分五层排查。

第一是 **模型输出层**：模型有没有选错工具、参数有没有幻觉、是否输出了不完整 tool call，或者 stop reason 是否异常。

第二是 **Agent Loop 层**：是否超过最大轮次、是否重复调用同一个工具、是否触发时间或 token 熔断。

第三是 **工具执行层**：工具参数格式是否正确，业务校验是否通过，权限是否允许，handler 是否超时，外部 API 是否失败。

第四是 **上下文层**：模型是否看到了正确的信息，工具结果是否被截断，上下文是否爆了，压缩是否丢失了关键任务状态。

第五是 **运行时层**：连接是否中断、任务状态是否持久化、多 Agent 是否有并发冲突、日志是否能还原现场。

这样排查的好处是不会把所有问题都归因给模型，而是能定位到底是模型决策问题、工具管线问题、上下文治理问题，还是 Harness 恢复能力问题。

---

### Q2：工具调用失败时，Agent 应该怎么处理？

**参考回答：**

工具调用失败不能简单把错误丢给模型再让它猜。应该先对失败做分类。

常见类型有：

- **格式错误**：JSON 解析失败、必填字段缺失、类型不匹配。
- **业务错误**：路径不存在、资源 ID 不存在、参数值不合法。
- **权限错误**：工具需要用户授权或当前 Agent 没有 capability。
- **执行错误**：handler 抛异常、外部 API 失败。
- **超时错误**：工具执行超过阈值。
- **结果错误**：返回过长、返回格式不符合预期。

处理策略也不同：

- 格式错误可以反馈给模型，让它修正参数后重试。
- 业务错误通常不应该盲目重试，而要换策略或询问用户。
- 权限错误要进入审批流程。
- 超时错误要返回结构化错误，避免卡死 Loop。
- 外部 API 临时失败可以指数退避，但不能无限重试。

核心原则是：模型输出不可信，工具执行前后都要经过 Tool Pipeline 校验和清洗。

---

### Q3：Tool Pipeline 为什么要区分格式校验和业务校验？

**参考回答：**

因为这两类错误的修复方式不同。

格式校验关注的是参数结构对不对，比如字段类型、必填字段、JSON 能否解析。这类错误通常可以告诉模型“字段 X 需要 number，不是 string”，让模型修正后重试。

业务校验关注的是语义是否合法，比如文件是否存在、用户是否有权限、时间范围是否合理、资源状态是否允许操作。这类错误不一定能靠重试解决，可能需要先查询状态、换工具，或者问用户。

如果把两者混在一起，模型不知道到底是格式错了，还是业务条件不满足，就容易重复犯错。

所以生产级 Tool Pipeline 通常会分层：格式校验 → 业务校验 → 权限校验 → 资源配额 → 副作用门控 → 执行 → 结果过滤。

---

### Q4：工具执行超时怎么办？

**参考回答：**

工具执行必须有超时保护，否则一个慢工具就会卡死整个 Agent Loop。

超时后不应该让请求一直挂着，而应该由 handler 返回结构化错误，例如：

```txt
status: failed_timeout
operation: search_documents
timeout_ms: 30000
partial_result: available / unavailable
safe_next_steps:
  - narrow query
  - retry once
  - ask user to reduce scope
```

然后 Loop 层根据错误类型决定是否重试。

如果是临时网络抖动，可以指数退避重试；如果是查询范围太大，就应该缩小范围；如果连续多次同样超时，就要触发重复失败检测或熔断。

关键点是：超时是一种工程容错机制，不是 bug。Agent 不能假设工具一定返回，也不能无限等待。

---

### Q5：工具返回内容太长怎么办？

**参考回答：**

工具结果是上下文膨胀的主要来源之一。比如搜索结果、日志、网页内容、文件内容都可能非常长。

处理方式是：

1. **实时截断**：超过阈值先截断，并显式标记 `...[truncated]`。
2. **摘要回写**：把关键结论、错误、路径、ID 保留在上下文里。
3. **完整结果落盘**：把完整日志或内容保存到文件，给模型一个引用路径。
4. **按需读取**：模型下一轮只读取相关片段，而不是一次性塞满上下文。

不能把截断结果伪装成完整结果，否则模型会基于不完整事实做错误判断。

这也是 Context Engineering 的核心思想：不是塞更多，而是在正确时间给正确的信息。

---

### Q6：如果 Agent 反复调用同一个失败工具，怎么办？

**参考回答：**

这是典型 Agent Loop 死循环问题。模型可能拿到失败结果后没有真正改变策略，而是不断重复调用。

课程里的处理思路是 Loop 检测与去重：

1. 对最近 N 步工具调用做结构或语义哈希，生成指纹。
2. 如果工具名、参数、错误摘要高度相似，就判定可能重复。
3. 分三级处理：
   - Warning：低置信度重复，提醒模型检查策略。
   - Critical：高置信度重复，注入提示要求换思路。
   - Break：明确死循环，直接熔断。

这比简单设置最大轮次更好。最大轮次只能兜底，而重复检测能更早发现“无进展”。

---

### Q7：为什么 Agent Loop 通常是 `while(true)`，但又必须有熔断？

**参考回答：**

因为 Agent 的工具调用轮次是动态的。模型每一轮会根据观察结果决定下一步，所以系统很难提前用 `for` 循环设定准确次数。

但 `while(true)` 不能真的无限跑，必须有外层保险丝，包括：

- **步数熔断**：超过最大轮次强制终止。
- **时间熔断**：超过最大墙上时间强制终止。
- **token 预算熔断**：上下文接近窗口或成本过高时停止或压缩。
- **重复调用熔断**：检测无进展循环。
- **模型终止信号**：根据 stop reason 判断正常结束或异常终止。

所以核心是：Loop 结构允许开放式探索，但 Harness 必须保证它可停止、可解释、可恢复。

---

### Q8：连接中断后，Agent 任务怎么恢复？

**参考回答：**

连接中断后不能只恢复聊天文本，还要恢复任务状态。

我会持久化这些信息：

- session / task ID。
- 当前用户目标。
- 已完成步骤和当前步骤。
- 最近工具调用及结果。
- 外部操作的幂等 ID。
- 权限审批状态。
- 当前上下文摘要。
- 如果有外部副作用，还要记录执行阶段。

恢复时不能直接继续模型上一次想做的动作，而应该先重新观察事实状态，比如工具资源是否已经变化、外部请求是否成功、任务是否已经完成。

如果历史摘要和当前观察冲突，应该信当前观察。因为记忆和 transcript 可能过期，而工具查询到的当前状态是一手事实。

---

### Q9：连接中断时，工具请求已经发出但结果未知，怎么办？

**参考回答：**

这是幂等和两阶段提交问题。

如果工具有副作用，比如发消息、创建订单、写数据库、调用外部 API，连接中断时可能出现“不知道是否执行成功”的状态。

不能盲目重试，否则可能重复执行。

可靠做法是：

1. 服务端生成 `retry_group_id` 或 idempotency key，不依赖模型的 `tool_call_id`。
2. 执行前记录预提交状态：要执行什么、参数是什么、目标资源是什么。
3. 工具执行后记录完成状态和外部结果 ID。
4. 恢复时先查询外部状态：已执行、未执行、不确定。
5. 根据状态决定返回结果、继续执行、回滚或请求人工确认。

核心信任边界是：不要相信模型自己不会重复调用，要相信服务端自己的幂等记录。

---

### Q10：上下文爆了怎么办？

**参考回答：**

上下文爆了不能只粗暴删除历史，否则模型可能丢失任务目标和关键约束。

推荐做渐进式压缩：

1. **先处理工具结果**：工具输出通常最大，先截断长结果，保留摘要、错误、路径和 ID。
2. **替换老旧 tool result**：老工具结果可以替换成占位符，保留结构痕迹。
3. **按轮次裁剪历史**：保留最近 N 轮用户意图和相关工具结果，避免切断半轮对话。
4. **LLM 摘要压缩**：最后才用模型生成结构化摘要。
5. **恢复当前工作集**：压缩后重新注入当前目标、计划、关键文件、失败原因、已尝试方案。

OpenClaw 这类系统会在 `transformContext` 阶段做压缩、裁剪、注入和最近轮保护。Claude Code 则有 Microcompact、Snip、Auto-compact 这类渐进策略。

一句话：上下文管理的目标不是少，而是让模型在正确时间看到正确信息。

---

### Q11：上下文压缩时最不能丢什么？

**参考回答：**

最不能丢的是“当前任务状态”和“失败经验”。

具体包括：

- 用户当前目标。
- 最近一次明确指令。
- 已完成步骤和下一步。
- 工具调用失败原因。
- 已经尝试过但失败的方案。
- 关键 ID、路径、文件名、URL、commit hash、任务 ID。
- 权限审批结果。
- 当前工作集，比如最近读过或修改过的文件。

如果压缩只保留了大概总结，却丢掉失败原因，模型会重复踩坑；如果丢掉关键 ID，恢复时就找不到资源。

所以结构化摘要要有质量守卫，比如检查固定 section 是否齐全、关键标识符是否保留、最近用户请求是否体现。

---

### Q12：为什么工具列表不能全部塞进上下文？

**参考回答：**

工具太多会挤占上下文窗口，也会稀释模型注意力，导致选错工具。

课程里提到工具数量变多后，模型选择准确率会明显下降。解决方式有两类：

- **按需注入工具子集**：根据当前意图选择相关工具，而不是全量注入。
- **Tool Profile / Deferred Tool Loading**：按场景预选工具，或者先用 ToolSearch 找到需要的工具 schema。

这里要区分注册表和上下文：

- 注册表是系统里所有工具的静态全集。
- 上下文是每次发给模型的工具子集。

好的 Tool System 不只是“工具越多越强”，而是让模型当前只看到真正相关的工具。

---

### Q13：模型 API 流式响应中断怎么办？

**参考回答：**

流式中断要先判断中断发生在哪个阶段。

如果是在普通文本输出阶段，可以保留已收到内容，提示用户中断并考虑继续生成。

如果是在 tool call 参数生成阶段，就比较危险，因为参数可能是不完整 JSON，不能直接执行。需要等参数完整、解析通过后才能执行工具。

如果工具已经开始执行，恢复时要看工具是否有副作用，使用幂等 ID 检查是否已经执行成功。

生产级策略包括：

- 服务端心跳和客户端看门狗。
- 流式失败后必要时降级非流式。
- 对已接收内容做状态标记。
- 对 tool call 参数做完整性校验。
- 对有副作用工具做幂等恢复。

核心原则：流式输出可以提升体验，但不能牺牲工具执行安全。

---

### Q14：模型 API 限流或服务不可用怎么办？

**参考回答：**

不能只写 try-catch，而要做错误分类和降级。

常见策略：

- 网络错误：短暂重试，指数退避。
- 限流错误：尊重 `Retry-After`。
- 服务端 5xx：有限重试，必要时切换 provider。
- 上下文过长：触发压缩后重试。
- 模型能力不足：降级或升级模型要看任务风险。
- 长时间失败：停止并告诉用户，而不是无限烧成本。

回答时可以强调：API 容错不是“失败就重试”，而是错误分类、预算控制、降级策略和用户可见反馈。

---

### Q15：Memory 和当前工具观察结果冲突，应该怎么办？

**参考回答：**

应该信当前观察。

Memory 是长期经验，可能过期；工具观察是当前事实，比如当前文件内容、数据库状态、接口返回、用户最新指令。

处理方式是：

1. 检测冲突。
2. 以当前观察为准继续任务。
3. 如果旧记忆会影响后续决策，就更新或标记过期。
4. 如果冲突涉及用户偏好或高风险决策，要向用户确认。

这类问题体现 Memory 不是绝对真理，而是可审查、可更新的辅助上下文。

---

### Q16：多 Agent 协作时，一个子 Agent 失败了怎么办？

**参考回答：**

先看子 Agent 失败类型。

如果是探索型子任务失败，比如没找到文件，可以让父 Agent 换搜索策略或派另一个子 Agent。

如果是写操作失败，要检查是否影响共享资源，必要时回滚或隔离。

Multi-Agent 的核心不是角色扮演，而是上下文隔离。子 Agent 应该在独立上下文里完成大量搜索或试错，最后只把压缩结论返回父 Agent。

父 Agent / Leader 要负责：

- 汇总子 Agent 结果。
- 判断是否可信。
- 统一处理权限请求。
- 控制写操作串行。
- 决定任务是否真正完成。

不能让 Worker 自己宣布全局完成，也不能让多个 Agent 同时写同一共享资源。

---

### Q17：为什么读工具可以并发，写工具要串行？

**参考回答：**

读工具通常没有副作用，比如搜索、读取文件、查询状态，可以并发提高效率。

写工具会改变外部状态，比如写文件、发消息、改数据库、提交任务。如果并发写同一个资源，可能产生竞态、覆盖、顺序错乱和难以审计的问题。

所以生产级 Agent 一般遵循：

- 读并发。
- 写串行。
- 高风险写操作需要权限审批。
- 共享资源写入要走队列或锁。
- 多 Agent 写操作由 Leader 统一协调。

这本质上是把传统后端并发控制思想引入 Agent Runtime。

---

### Q18：Agent 执行了危险工具怎么办？如何预防？

**参考回答：**

预防比事后补救更重要。

工具执行前要有权限和风险分类：

- allow：低风险自动允许。
- deny：明确禁止。
- ask：高风险需要用户确认。

例如删除文件、发外部消息、支付、改生产数据、执行系统命令都应该进入 ask 或 deny。

工具还要有沙箱、路径白名单、资源配额和 Hook。Hook 可以在 PreToolUse 阶段阻止风险操作，在 PostToolUse 阶段记录审计日志。

如果已经执行了危险工具，要立即停止 Loop，记录 transcript，汇报影响范围，并提供回滚或补救方案。

---

### Q19：如何让 Agent 失败后用户能看懂发生了什么？

**参考回答：**

失败输出要结构化，而不是直接贴一大段日志。

可以包含：

```txt
失败阶段：工具执行 / 上下文压缩 / API 调用 / 权限审批
失败工具：xxx
错误类型：timeout / permission_denied / invalid_params / context_overflow
已尝试方案：...
当前状态：可重试 / 需要用户输入 / 已熔断
建议下一步：...
日志引用：...
```

这样用户知道 Agent 卡在哪里，也方便后续恢复。

另外系统内部要有 JSONL transcript 或事件日志，记录用户输入、模型输出、工具调用、工具结果、权限事件和异常状态。这是可观测性的基础。

### Q20：请设计一个生产级 Agent 失败恢复架构。

**参考回答：**

我会设计成六个模块。

1. **Agent Loop Controller**：负责 Observe → Think → Act，多轮调用，并提供步数、时间、token、重复调用熔断。
2. **Tool Pipeline**：负责格式校验、业务校验、权限、资源配额、副作用门控、执行和结果过滤。
3. **Context Manager**：负责工具结果截断、历史裁剪、结构化摘要、记忆注入和当前工作集恢复。
4. **State Manager**：持久化 task ID、session、当前步骤、工具调用、幂等 ID、权限状态和外部资源 ID。
5. **Recovery Manager**：连接中断或进程重启后，重新观察事实状态，判断已完成、未完成或不确定。
6. **Observability & Harness**：提供事件日志、Hook、沙箱、权限审批、成本预算、多 Agent 并发控制。

最后我会强调：生产级 Agent 的核心不是模型一次回答多聪明，而是在失败、超时、上下文爆掉、连接断开时，系统仍然能安全停下、解释原因，并从正确状态恢复。

## 3. 高频追问速答

### 追问 1：Agent 工具调用失败，第一步做什么？

先分类错误：格式、业务、权限、超时、外部 API、结果过长。不同错误采用不同策略，不要无脑重试。

### 追问 2：什么时候可以让模型重试？

格式错误、临时网络错误、可恢复的 API 错误可以有限重试；业务错误和权限错误不应该盲目重试。

### 追问 3：为什么要用 `retry_group_id`，不用 `tool_call_id`？

`tool_call_id` 可能由模型或协议生成，重试时会变；`retry_group_id` 是服务端业务级幂等 ID，更可信。

### 追问 4：上下文爆了最先压缩什么？

先压缩工具结果，因为工具输出通常是最大噪声源。

### 追问 5：压缩后最容易出什么问题？

丢失用户目标、失败原因、关键 ID、权限状态和当前工作集，导致模型重复踩坑或状态漂移。

### 追问 6：模型一直重复同一个工具怎么办？

做调用指纹检测，按 Warning / Critical / Break 分级处理，必要时熔断。

### 追问 7：连接中断恢复时信历史还是信当前查询？

信当前查询。历史和记忆可能过期，当前工具观察是一手事实。

### 追问 8：工具返回被截断后能直接下结论吗？

不能。必须标记 truncated，并提供完整结果引用或继续读取机制。

### 追问 9：多 Agent 的核心价值是什么？

不是角色扮演，而是上下文隔离、任务隔离和风险隔离。

### 追问 10：Harness 解决什么问题？

Harness 是模型外面的运行环境，负责权限、Hook、日志、沙箱、调度、状态恢复、上下文压缩和可观测性。

## 4. 两分钟背诵版

如果面试官问 Agent 运行失败怎么处理，我会先说：Agent 失败不是单点 bug，而是模型、工具、上下文、状态和运行时共同作用的问题，所以要从 Agent Runtime 角度设计容错。

第一，Agent Loop 要有保险丝。因为工具调用轮次不可预设，Loop 通常是开放式的，但必须有步数熔断、时间熔断、token 预算控制和重复调用检测。如果模型反复调用同一个失败工具，要对最近 N 步做指纹检测，按 Warning、Critical、Break 分级处理。

第二，工具调用要走 Tool Pipeline。模型输出不可信，只代表意图。执行前要做格式校验、业务校验、权限校验、资源配额和副作用门控；执行时要有超时；执行后要做结果过滤、脱敏、截断和结构化错误回写。格式错误可以让模型修正，业务错误和权限错误不能无脑重试。

第三，上下文爆了要做 Context Engineering。不要粗暴删除历史，而是先压缩工具结果，再替换老旧 tool result，再按轮次裁剪，最后才用 LLM 摘要。压缩后必须恢复当前工作集，包括用户目标、失败原因、已尝试方案、关键 ID、权限状态和下一步。

第四，连接中断要靠状态持久化和幂等恢复。对于已经发出的工具请求，不能盲目重试，要用服务端生成的 retry_group_id 记录预提交状态，恢复后重新观察外部事实，判断已完成、未完成还是不确定。

最后，生产级 Agent 还需要 Harness，把权限、Hook、审计日志、沙箱、多 Agent 并发控制和恢复机制放在模型外面。核心思想是：不要假设模型会停、会省 token、会选对工具、会保护权限，这些都要靠系统工程兜底。

## 5. 关键词速记

- Agent Loop：Observe → Think → Act、while(true)、步数/时间/token 熔断
- Loop 检测：调用指纹、Warning / Critical / Break
- Tool Pipeline：格式校验、业务校验、权限、资源配额、副作用门控、执行、结果过滤
- 超时保护：handler timeout、结构化错误、有限重试
- 结果截断：truncated 标记、摘要回写、完整日志落盘
- 幂等恢复：retry_group_id、两阶段提交、当前事实优先
- Context Engineering：Microcompact、Snip、Auto-compact、transformContext
- 上下文恢复：当前目标、失败原因、已尝试方案、关键 ID、权限状态
- Memory 冲突：记忆和当前观察冲突时信当前观察
- Harness：权限、Hook、JSONL transcript、沙箱、调度、可观测性
