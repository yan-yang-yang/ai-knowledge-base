# Agent 架构模式

> 来源：`/root/ai-text/2/`、`/root/ai-text/4/`、`/root/ai-text/5/`、`/root/ai-text/6/` Sitor「吃透 AI Agent 开发」课程材料。  
> 用途：沉淀 Agent 工程里可复用的架构模式，而不是绑定某个框架 API。

---

## 0. 一句话总结

**Agent 架构模式的核心是围绕不确定的模型输出构建确定性的工程边界：循环要有保险丝，工具要有管线，状态要能恢复，上下文要能治理，多 Agent 要为隔离服务，Harness 要提供安全和观测。**

---

## 1. Agent Loop 保险丝模式

课程在 Agent Loop 章节强调三根保险丝：

| 保险丝 | 解决问题 | 典型机制 |
|---|---|---|
| 工具死循环检测 | 重复调用同一工具、重复犯错 | 调用指纹、检测器、分级响应 |
| Token 预算控制 | 上下文和成本烧穿 | 预算估算、提前拦截、压缩或收尾 |
| 输出截断恢复 | 模型输出没说完 | 提高输出上限、继续生成、恢复次数限制 |

退出路径也要明确，包括：

- 正常完成
- 超过最大轮次
- 用户中断
- 工具执行中断
- Hook 停止
- 上下文接近上限
- prompt 过长但可恢复

## 2. 流式响应与边说边执行

生产级 Agent 不能让用户长时间看空白。

课程总结的流式要点：

- 模型输出天然是 token 流。
- Tool Call 的参数可能是碎片，需要攒完整再执行。
- 工具执行和文本输出可以形成“边说边执行”的体验。
- 不是所有工具都能提前执行，写操作和需要审批的操作要谨慎。
- 不同模型提供商的流式协议有差异，需要适配层。

这个模式的价值：

- 降低感知延迟。
- 让用户看到 Agent 正在做什么。
- 为审批和中断提供交互点。

## 3. API 容错与降级模式

模型 API 失败不能只加 try-catch。

课程给出的分层决策：

1. 错误分类：网络错误、限流、服务端错误、上下文过长等。
2. 单次失败重试：指数退避，尊重 Retry-After。
3. 流式卡住：服务端心跳 + 客户端看门狗。
4. 流式失败：必要时降级为非流式。
5. 模型降级：高能力模型不可用时切换到更稳模型。
6. 多 Provider 容灾：区分临时性和持久性故障。

核心原则：

- 不要无限重试。
- 已接收内容要有处理策略。
- 用户应该知道发生了什么。

## 4. 上下文压缩模式

课程对比了 Claude Code 和 OpenClaw 的压缩策略。

Claude Code：

1. Microcompact：先压缩工具结果。
2. Snip：裁剪旧消息，同时修复 tool_use/tool_result 配对。
3. Auto-compact：最后用 LLM 摘要。

OpenClaw：

1. Tool Result Context Guard：工具结果实时截断。
2. History Turn Limiting：历史轮次裁剪。
3. Compaction Safeguard：LLM 结构化摘要。
4. Overflow Compaction：溢出后重试。

共同原则：

- 先轻后重。
- 先处理最大噪声源：工具结果。
- 最近轮和当前任务要保护。
- 错误和修复方式要保留。

---

## 5. Multi-Agent 上下文隔离模式

课程明确反对“为了分角色而分 Agent”。

Multi-Agent 更实际的价值：

- 隔离大范围探索。
- 隔离高风险修改。
- 隔离长任务状态。
- 并行处理互不依赖的任务。

三种隔离思路：

| 思路 | 机制 | 适合场景 |
|---|---|---|
| 共享内存/进程内隔离 | AsyncLocalStorage 等 | 轻量子任务、低延迟探索 |
| Worktree 隔离 | 独立 git worktree | 高风险代码修改、重构实验 |
| 队列串行化 | 用排队代替锁 | 多任务共享资源、避免写冲突 |

---

## 6. Swarm 团队协作模式

Agent Swarm 比父子 Agent 更复杂，多了团队协作能力。

课程提到的关键组件：

- Mailbox：Agent 间消息系统。
- 共享任务列表：防止抢活和重复工作。
- Permission Sync：Worker 权限请求同步给 Leader / 用户。
- Leader 审批关闭：Worker 不能自己决定“我完成了”。
- 多执行后端：进程内、tmux、iTerm2 等。

适用条件：

- 任务足够复杂。
- 子任务相对独立。
- 有明确协调机制。
- 权限和状态同步成本可接受。

---

## 7. Harness 模式

Harness 是模型外面的壳。

课程里的 Harness 零件包括：

- 权限系统
- Hook
- 可观测性
- 沙箱
- 调度
- 状态持久化
- 成本监控
- 上下文压缩
- Memory
- Sub-agent

经典 Harness 模式：Generator / Evaluator。

- Generator：生成候选方案。
- Evaluator：检查、打分、拒绝或要求重试。

这个模式适合代码生成、测试修复、文档生成等任务。

## 9. 面试表达

### 9.1 两分钟回答版

> 我理解 Agent 架构模式的核心，是在模型不确定输出外面建立确定性的工程边界。Agent Loop 要有保险丝，比如死循环检测、Token 预算、输出截断恢复和明确退出路径；流式响应要支持边说边执行，但工具参数要攒完整，高风险工具要审批；模型 API 要有错误分类、指数退避、Retry-After、流式降级、模型降级和多 Provider 容灾。上下文方面要先处理工具结果，再裁剪历史，最后才用 LLM 摘要，并保护最近任务和错误修复路径。Multi-Agent 不是分角色，而是分上下文，可以用子 Agent、worktree 或队列隔离探索和写冲突。最后，Harness 把权限、Hook、观测、沙箱、调度和状态恢复包起来，保证 Agent 能在生产环境可控运行。

### 9.2 高频追问

#### Q1：为什么 Agent 必须有退出路径？

因为模型不一定会自然停下来，工具调用也可能循环。退出路径是系统层安全边界，不能完全依赖模型自觉。

#### Q2：什么时候值得用 Swarm？

只有当任务复杂、子任务可并行、协调收益大于通信和权限同步成本时才值得用。否则父子 Agent 或单 Agent 更简单。

#### Q3：Harness 会不会随着模型变强消失？

课程观点是 Harness 会变薄，但不会完全消失。因为权限、安全、审计、调度、成本控制这些是工程系统责任，不是模型能力本身能完全替代的。
