# RAG、检索优化与 LLM 编译知识库

> 来源：`/root/ai-text/4/` Sitor「吃透 AI Agent 开发」课程材料。  
> 用途：理解从文档到 Agent 可用知识库的完整链路，以及为什么“语义相似”不等于“任务相关”。

---

## 0. 一句话总结

**RAG 的核心不是“把文档丢进向量库”，而是围绕加载、分块、Embedding、存储、检索、重排、更新和 Agentic 使用构建一条可持续维护的知识管线。**

---

## 1. RAG 全流程

课程把 RAG 拆成几个关键步骤：

| 步骤 | 目标 | 常见坑 |
|---|---|---|
| 文档加载 | 把 PDF、Markdown、网页、代码等转为可处理文本 | 格式丢失、噪声过多、元数据缺失 |
| 分块 | 把长文档切成可检索片段 | 切太碎丢语义，切太大召回不准 |
| Embedding | 把文本变成向量 | 模型选型、中文效果、成本 |
| 存储 | 存向量、文本和元数据 | 只存向量导致难审计 |
| 检索 | 根据 query 找候选片段 | 语义相似但任务不相关 |
| 索引更新 | 文档变化后重建或增量更新 | 旧索引污染结果 |

Agent 场景下，RAG 不只是问答系统的检索模块，而是 Agent Loop 中的一个工具：模型可以多轮查询、改写 query、读取结果、继续检索。

---

## 2. 分块：检索质量的第一道门

课程强调：分块切得好不好，直接决定检索质量。

常见策略：

- 固定长度分块
- 按标题/段落结构分块
- 语义分块

代码文件还要考虑：

- 函数边界
- 类边界
- import / export
- 调用关系

一个重要判断：**分块不是越小越好。**

- 太小：上下文不足，模型看不懂。
- 太大：召回噪声多，浪费上下文。

---

## 3. 检索优化：语义相似不等于任务相关

纯向量搜索有结构性局限：它擅长找“语义相近”，但用户真正需要的是“能完成当前任务的信息”。

课程给出几类优化：

### 3.1 混合检索

结合：

- 向量搜索
- 关键词搜索 / BM25

OpenClaw 的记忆检索就是混合思路：向量负责语义，关键词负责精确术语、路径、名称等。

### 3.2 Query 改写

用户问法不一定等于最佳检索 query。

常见方式：

- HyDE：先生成一个假设答案/文档，再用它去 embedding。
- 子问题分解：把复杂问题拆成多个检索问题。

### 3.3 Reranker

粗召回后再精排。

- Bi-encoder：适合快速召回。
- Cross-encoder：相关性判断更强，但成本更高。

所以通常不是直接全量 Cross-encoder，而是先召回候选，再精排。

### 3.4 MMR 去重

如果搜回来 6 条，5 条都说同一件事，信息增量很低。MMR 用来平衡相关性和多样性。

### 3.5 时间衰减

旧信息不一定错，但在项目知识库和记忆系统里，较新的信息通常应该更优先。

## 4. 从 RAG 到 Agentic RAG

传统 RAG 常是一次性流程：

```txt
query → retrieve → answer
```

Agentic RAG 更像循环：

```txt
observe → decide query → retrieve → inspect → refine query → retrieve again → answer / act
```

课程在 JIT Context 里也提到三条路线：

| 路线 | 机制 | 适合场景 |
|---|---|---|
| Agentic Search | Glob / Grep / Read 等确定性搜索 | 代码仓库、结构化文件 |
| RAG | 向量 + 关键词 + rerank | 大量非结构化文档 |
| Context Offloading | 把中间状态放到文件系统等外部载体 | 长任务、可恢复上下文 |

所以 RAG 不是所有检索问题的唯一答案。代码场景下，确定性搜索经常比向量搜索更可靠。

---

## 5. LLM 编译知识库

课程提出一个重要方向：RAG 的尽头可能是让 AI 帮你“写书”，即 LLM 编译知识库。

它把知识库类比成编译系统：

| 层级 | 作用 |
|---|---|
| raw/ | 原始素材，保留来源 |
| wiki/ | 编译产物，结构化、互联、可读 |
| schema | 操作规范，约束如何摄入、查询、清理 |

三个核心操作：

- Ingest：摄入新资料。
- Query：查询和使用知识。
- Lint：清理重复、过期、冲突内容。

这和当前知识库的实践很接近：`/root/ai-text` 是 raw，`knowledge-base/agent/` 是 wiki，`agent-article-distiller` skill 是 schema / workflow。

## 6. 面试表达

### 6.1 两分钟回答版

> RAG 不是简单把文档切块丢进向量库，而是一条完整知识管线。首先要加载文档并保留结构和元数据，然后选择合适的分块策略；分块太小会丢上下文，太大又会召回噪声。接着用 Embedding 建向量索引，同时最好保留原文和元数据，方便审计和重建。检索阶段不能只依赖语义相似，因为语义相似不等于任务相关，所以生产系统通常会做混合检索，把向量和关键词结合，再通过 query 改写、HyDE、子问题分解、reranker、MMR 去重和时间衰减提升结果质量。到了 Agent 场景，RAG 会变成 Agentic RAG：模型可以在 Loop 中多轮检索、检查结果、调整 query。再往前一步，就是 LLM 编译知识库，把 raw 原始素材编译成 wiki 结构化知识，并通过 schema 约束持续维护。

### 6.2 高频追问

#### Q1：为什么不能只用向量检索？

因为向量检索擅长语义相似，但很多任务需要精确匹配，比如 API 名称、文件路径、错误码、配置项。关键词/BM25 能补上这类精确召回能力。

#### Q2：RAG 和 Agentic Search 怎么选？

如果资料是大量自然语言文档，RAG 更合适；如果是代码仓库、配置文件、结构化目录，Glob/Grep/Read 这种确定性搜索经常更准；复杂 Agent 可以把两者都作为工具。

#### Q3：LLM 编译知识库和普通 RAG 的差别？

普通 RAG 更像运行时检索；LLM 编译知识库会把原始素材整理成结构化、互联、可维护的知识产物，更适合长期积累和复习。
