# 2026-07-21：RAG 知识库如何从 0 搭建，以及如何优化？

> 题目来源：用户每日面试题。  
> 原始题意：关于 RAG 知识库，包括刚开始如何切分，后面如何优化。  
> 依据：Sitor「吃透 AI Agent 开发」课程 RAG 全流程、检索优化、LLM 编译知识库，以及公共雨伞项目讲稿。

---

## 1. 面试官真正想问什么？

这道题表面是在问 RAG 怎么做，实际考的是你有没有完整的 RAG 工程链路意识。

面试官不是想听：

> 文档切片、embedding、存向量库、检索、丢给模型。

这个太浅了。

他真正想听的是：

1. **知识库从 0 到 1 怎么搭**：文档加载、清洗、切分、metadata、embedding、入库、检索、上下文注入。
2. **切分策略怎么选**：固定切分、递归切分、语义切分、结构化切分，各自适合什么。
3. **为什么检索不准**：语义相似不等于任务相关。
4. **怎么优化**：metadata 过滤、混合检索、query 改写、reranker、MMR、时间衰减、评估指标。
5. **怎么结合业务**：不是泛文档问答，而是围绕业务意图、规则版本、SOP 完整性来设计。
6. **什么时候不用 RAG**：小知识库可以直接上下文；代码检索更适合 Agentic Search；知识持续积累可以考虑编译知识库。

---

## 2. 1 分钟回答

RAG 知识库我会分两阶段讲：先搭通，再优化。

从 0 到 1 时，第一步是文档加载和清洗，把 Markdown、PDF、网页、Word 等不同格式转成干净文本。第二步是分块，默认我会从递归分块开始，优先按标题、段落、句子切，chunk size 大概 256-512 token，不会一上来用语义分块，因为语义分块容易切得太碎。第三步给 chunk 加 metadata，比如业务类型、文档版本、适用场景、权限级别。第四步做 embedding，存入向量数据库，再根据用户问题检索 top-k chunk，拼进上下文生成答案。

优化时，核心问题是“语义相似不等于任务相关”。所以不能只靠纯向量搜索。我会先做意图识别和 metadata 过滤，再做混合检索，把向量搜索和 BM25 关键词搜索结合；复杂问题可以做 query 改写或子问题拆解；召回后用 reranker 精排；结果重复时用 MMR；有时间敏感内容时加时间衰减。最后用召回命中率、答案准确率、引用可信度、幻觉率和业务处理效率来评估。

在公共雨伞业务里，我不会把所有文档简单丢进向量库，而是按借伞、归还、押金、设备、网点、活动、售后 SOP 组织知识库，让检索服务业务判断。

## 3. RAG 从 0 到 1 怎么做？

### 3.1 文档加载：先保证数据质量

RAG 第一步不是 embedding，而是把文档变成干净、可用的文本。

不同格式处理方式不一样：

- **Markdown**：最友好，天然有标题、列表、代码块，结构信息保留好。
- **PDF**：最容易出问题，普通文本抽取可能把表格、多栏、页眉页脚搞乱，需要结构化解析工具。
- **网页**：要去掉导航栏、广告、footer、cookie 提示等噪声。
- **代码文件**：不能按普通文章处理，最好按函数、类、模块等结构切。
- **业务文档 / SOP**：要保留完整步骤，不能机械切断。

课程里有一个关键观点：**数据质量决定 RAG 上限。** 后面 embedding 模型再好，也救不了解析混乱的原始文本。

### 3.2 分块：切得好不好，直接决定检索质量

分块是 RAG 里非常关键的一步。因为上下文窗口有限，不可能把整份文档都塞给模型，只能检索最相关的 chunk。

常见分块策略：

1. **固定大小分块**  
   按 token 数硬切，比如每 512 token 一块。简单，但容易把一句话或一个 SOP 步骤切断。

2. **递归分块**  
   先按标题 / 段落切，太长再按句子，最后按 token。它尽量保留语义完整性，是大多数场景的稳妥起点。

3. **语义分块**  
   用 embedding 判断相邻段落语义变化，在语义变化大的地方切。听起来高级，但实际可能切得太碎，chunk 上下文不足。

4. **结构化分块**  
   按标题层级、页面、章节、SOP 步骤、代码 AST 等结构切。

我的默认选择是：

> 文档类知识库先用递归分块，chunk size 256-512 token；业务 SOP 尽量按完整流程切；代码类内容优先用 AST 或 Agentic Search，不优先做向量 RAG。

### 3.3 Metadata：让检索带业务约束

RAG 不能只靠语义相似度。业务知识库一定要加 metadata。

比如公共雨伞业务里，我会给 chunk 加：

```txt
business_type: borrow / return / payment / device / station / activity / after_sales
intent: return_order_not_closed / borrow_failed / refund_policy
version: 规则版本
city: 适用城市
visible_to: customer_service / operation / internal
need_human_review: true / false
updated_at: 更新时间
```

这样用户问“还伞后订单还在计费”，系统可以先过滤 `return` 和 `after_sales` 相关文档，避免召回优惠活动或借伞规则。

### 3.4 Embedding 和向量库

分块后把 chunk 转成 embedding 向量，再存入向量数据库。

选型上我会这样讲：

- 快速起步：OpenAI text-embedding-3-small 这类 API 模型；
- 中文 / 跨语言 / 长文本要求高：选更强 embedding 模型；
- 本地小模型要谨慎，特别是中文和长文本场景；
- 向量数据库不是质量核心，别过度纠结。

向量库选择：

- 本地 / 边缘：sqlite-vec；
- 原型验证：ChromaDB；
- 已有 PostgreSQL：pgvector；
- 大规模生产：Qdrant / Pinecone / Milvus。

### 3.5 检索和上下文注入

基础流程是：

```txt
用户问题
→ 意图识别 / Query 预处理
→ 向量检索 top-k
→ 取回 chunk
→ 拼进 Prompt
→ LLM 生成答案
```

但拼上下文时要注意：

- chunk 不宜过多，避免上下文污染；
- 要带来源，方便追溯；
- SOP 类内容要保留完整步骤；
- 冲突规则要标注版本和更新时间；
- 高风险回答要提示人工复核。

## 4. RAG 为什么会不准？

核心原因是：

> 语义相似 ≠ 任务相关。

比如用户问：

> 怎么部署这个服务？

纯向量搜索可能召回“服务架构概览”，因为它语义上和部署相关。但用户真正需要的是部署命令、环境变量、配置文件。

再比如用户问：

> 上次上线出了什么问题？

向量搜索可能召回“上线流程规范”，因为语义相近，但用户要的是事故复盘或错误日志。

所以生产级 RAG 不能只靠纯向量搜索，需要做检索优化。

## 5. RAG 怎么优化？

### 5.1 Metadata 过滤：先缩小业务范围

优化第一步不是上复杂模型，而是先用业务信息过滤。

比如公共雨伞业务：

- 用户问押金 → `payment`；
- 用户问还伞后计费 → `return + after_sales`；
- 运营问设备离线 → `device + station`；
- 活动规则 → `activity`。

先过滤业务类型，再做向量检索，能显著减少召回错场景。

### 5.2 混合检索：向量 + BM25

纯向量搜索懂语义，但对错误码、API 名、订单状态、专有名词不一定准。

BM25 关键词搜索不懂语义，但精确匹配很强。

所以生产常用混合检索：

```txt
向量搜索：理解语义
BM25：精确匹配关键词 / 错误码 / 专有名词
最终分数 = 0.7 × 向量分 + 0.3 × 关键词分
```

注意：向量分和 BM25 分不是同一个量纲，合并前要归一化。否则直接加权没有意义。

### 5.3 Query 改写：让问题更适合检索

用户的问题不一定适合直接搜索。

优化方式包括：

1. **HyDE**：先让模型生成一个“假回答”，再用假回答去检索。因为假回答的表达方式更像文档内容。
2. **子问题拆解**：复杂问题拆成多个 query 分别检索。
3. **关键词扩展**：把业务别名、错误码、同义词补进去。

例如：

> 用户说还伞后还在扣费怎么办？

可以改写成：

```txt
归还后订单未结束 SOP
订单仍计费 归还记录 设备离线
return_order_not_closed after_sales
```

### 5.4 Reranker：召回后再精排

Embedding 是 Bi-encoder：query 和 chunk 分别算向量，再比距离，快但不够精。

Reranker 通常是 Cross-encoder：把 query 和 chunk 放在一起判断相关性，慢但更准。

生产里一般用两阶段架构：

```txt
第一阶段：向量 + BM25 快速召回 20-50 条
第二阶段：Reranker 对候选精排，取 top-k
```

课程里强调：不加 Reranker 的 RAG，在生产环境里经常不够稳定。

### 5.5 MMR：避免结果重复

有时 top-k 结果都在讲同一件事，比如 6 条结果里 5 条都是同一个 SOP 的不同版本。

MMR 解决的是结果多样性：

```txt
MMR = 相关性 - 与已选结果的相似度惩罚
```

它能让结果既相关，又不要太重复。

### 5.6 时间衰减：让新信息优先

如果知识库有时间敏感性，比如活动规则、版本说明、会议记录、事故复盘，就要考虑时间衰减。

同样相关的两条内容，昨天更新的规则应该排在三个月前规则前面。

但基础制度、长期有效规则不一定要衰减，所以时间衰减要按文档类型配置。

### 5.7 索引更新：文档变化后怎么办

RAG 知识库不是一次性建设。

常见策略：

- **全量重建**：简单，但成本高，适合文档量小或 embedding 模型升级。
- **增量更新**：用 hash 检测文档是否变化，只处理变更部分。
- **临时库 + 原子交换**：全量重建时先构建临时索引，成功后替换旧索引，避免线上半成品状态。

如果 embedding 模型升级了，新旧向量不能混用，必须全量重建。

## 6. 怎么结合公共雨伞项目讲？

可以这样讲：

> 在公共雨伞业务里，我不会把所有文档简单切片丢进向量库，而是先按业务场景建知识库。比如借伞、归还、押金支付、设备异常、网点运营、活动规则、售后 SOP。切分时，普通说明文档用递归分块，chunk 大概 256-512 token；SOP 文档按完整处理流程切，保证模型拿到的是可执行步骤；活动规则会加版本和适用城市；客服话术会标记是否可直接对用户展示。  
> 检索时先做意图识别和 metadata 过滤，比如“还伞后仍计费”只检索 return 和 after_sales 场景，再做向量检索和关键词混合检索。如果召回结果不准，再加 query 改写、reranker 和规则版本过滤。最终输出时会带证据来源、风险等级和是否需要人工复核。这样 RAG 不是一个泛问答系统，而是服务业务异常处理的知识检索层。

## 7. 高频追问

### Q1：RAG 最容易失败在哪里？

最容易失败在数据质量和检索环节。文档解析乱、chunk 切坏、metadata 缺失、纯向量检索召回错场景，都会导致模型拿错资料，然后生成错误答案。

### Q2：chunk size 怎么选？

默认从 256-512 token 起步。太小会丢上下文，太大召回不精准且浪费 token。SOP 类文档要按完整步骤切，不要硬切断流程。

### Q3：语义分块是不是一定比递归分块好？

不是。课程里提到语义分块看起来高级，但实测可能切得太碎，导致 chunk 上下文不足。递归分块反而是大多数场景的稳妥起点。

### Q4：为什么不能只用向量检索？

因为语义相似不等于任务相关。向量搜索不擅长精确错误码、API 名、专有名词和操作性需求。生产里通常要结合 BM25、metadata、reranker。

### Q5：Reranker 为什么有用？

向量检索是快速召回，Reranker 是精排。Cross-encoder 能同时看到 query 和 chunk，更准确判断相关性，适合从 20-50 条候选里选真正相关的 top-k。

### Q6：RAG 和长上下文模型怎么选？

小文档库可以直接塞上下文，省掉 RAG 成本。但规模变大后，长上下文会注意力分散、成本高，RAG 的价值是只把最相关内容放进上下文。

### Q7：代码知识库适合 RAG 吗？

不一定。代码有明确结构，很多 Coding Agent 更倾向用 Grep / Glob / Read 做 Agentic Search。代码如果要切，最好按 AST 的函数、类、模块切，而不是普通 token 分块。

### Q8：知识库文档更新怎么办？

小规模可全量重建；大规模用 hash 做增量更新。embedding 模型升级时必须全量重建，因为新旧向量不在同一个空间。

### Q9：怎么评估 RAG 效果？

看召回命中率、答案正确率、引用准确率、幻觉率、人工复核率、用户问题解决率。业务系统里还要看客服处理时长是否下降、SOP 命中率是否提升。

### Q10：RAG 的尽头是什么？

不是无限优化向量库，而是让知识能持续积累。对于中小规模知识体系，可以考虑 LLM 编译知识库：raw 原文只读，wiki 是 LLM 维护的结构化知识，schema 定规范；查询结果有价值时反写回知识库。

## 8. 背诵版

RAG 知识库我会按“先搭通，再优化”来讲。先搭通时，第一步是文档加载和清洗，不同格式处理方式不一样，Markdown 最友好，PDF 要注意表格、多栏和页眉页脚，网页要去噪，SOP 要保留完整步骤。第二步是分块，默认我会用递归分块，从标题、段落、句子逐级切，chunk size 大概 256-512 token；不会一上来用语义分块，因为它可能切得太碎。第三步是加 metadata，比如业务类型、规则版本、适用城市、是否客服可见、是否需要人工复核。第四步是 embedding、向量入库、检索 top-k，再把结果带来源拼进上下文。

优化时，核心问题是语义相似不等于任务相关。纯向量搜索可能搜到语义近但不解决问题的内容。所以我会先做意图识别和 metadata 过滤，再用混合检索，把向量搜索和 BM25 结合；复杂问题做 query 改写，比如 HyDE 或子问题拆解；召回后用 reranker 精排；结果重复时用 MMR；有时间敏感的规则和活动时加时间衰减；文档更新时用 hash 做增量更新，embedding 模型升级时全量重建。

在公共雨伞业务里，RAG 不是泛文档问答，而是业务知识检索层。我会按借伞、归还、押金、设备、网点、活动、售后 SOP 来组织知识库。比如“还伞后仍计费”，先识别为 return_order_not_closed，再过滤 return 和 after_sales 文档，检索 SOP 和话术，最后结合订单、设备工具结果生成有证据、有风险等级、可人工复核的处理建议。
