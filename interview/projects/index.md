## 二、项目一：AI 智能业务运营与自动化系统

### 1. 面试官问：介绍一下你这个 AI 业务运营系统

#### 30 秒版本

这是一个面向业务运营场景的 AI 应用系统，主要解决信息查询、内容生成、流程处理和自动化辅助这些重复工作。系统后端用 FastAPI，AI 能力包括 LLM API、RAG 检索、Agent Workflow 和 SSE 流式响应。我的重点是把模型能力工程化：用 Pydantic 约束输入输出，用 LangGraph 管理任务状态和工具调用，用 RAG 提供业务知识上下文，再通过 SSE 把中间状态和最终结果实时返回前端。

#### 2 分钟版本

这个项目的背景是业务运营里有很多重复性问题，比如查业务资料、生成运营内容、根据规则处理流程、辅助判断下一步动作。如果完全靠人工，会比较耗时；如果只是简单接一个 Chat API，又很难保证回答基于业务知识，也没法和业务流程结合。

所以我把系统拆成几层：

1. **接口层**：用 FastAPI 提供任务提交、会话请求、状态查询和流式响应接口。
2. **模型调用层**：封装 LLM API，统一处理超时、重试、异常兜底和日志记录。
3. **RAG 检索层**：对业务文档做切分、Embedding、向量检索、Metadata 过滤和上下文拼装，让模型回答有依据。
4. **Agent 工作流层**：用 LangGraph / LangChain 建模任务状态，比如理解用户问题、判断是否需要检索、调用工具、生成结果、回写状态。
5. **前端交互层**：通过 SSE 推送模型生成内容、工具执行状态和最终结果，前端可以展示加载态、中间态、完成态和异常态。

这个项目我最大的收获是理解了 AI 应用不是简单调模型，而是要把模型放到一个可控的工程系统里：输入要校验，输出要约束，检索要可解释，工具调用要有权限和兜底，流式响应要能处理异常中断。

### 2. 面试官问：这个项目的核心难点是什么？

我会说核心难点有三个：

第一是 **模型输出不稳定**。LLM 可能格式不一致、答非所问，或者生成不符合业务要求的内容。我的处理方式是用 Prompt 明确任务边界，再用 Pydantic / JSON Schema 做结构化输出约束，必要时对模型输出做二次校验和后处理。

第二是 **RAG 检索质量**。如果召回内容不准，模型回答就会偏。这里需要处理文档切分粒度、Embedding 召回、关键词补充、Metadata 过滤、TopK 控制和上下文拼装顺序。我的思路是先保证召回内容和问题相关，再控制上下文长度，避免把无关内容塞给模型。

第三是 **Agent 流程可控性**。Agent 如果没有边界，可能重复调用工具、一直循环或在异常时无法收敛。所以要有状态流转、最大步数、超时控制、异常分支和兜底回答，而不是完全放任模型自由行动。

### 3. 面试官问：你为什么用 LangGraph，而不是只用 LangChain Chain？

因为这个项目不是单轮固定链路，而是有明显的状态流转和分支判断。

普通 Chain 更适合线性流程，比如“检索 → 拼 Prompt → 生成回答”。但业务运营任务里会出现：

- 是否需要知识库检索；
- 是否需要调用工具；
- 工具调用失败后是否重试；
- 结果是否需要结构化校验；
- 不同任务类型走不同节点。

LangGraph 的优势是可以把流程建模成节点和边，每个节点负责一个明确动作，状态在节点之间传递。这样流程更可控，也方便加重试、异常边、人工兜底和日志观测。

### 4. 面试官问：RAG 具体怎么设计？

我一般按五步设计：

1. **文档处理**：把业务文档、FAQ、规则说明等统一清洗成纯文本或 Markdown。
2. **分块**：按标题、段落和语义边界切分，避免块太大导致召回不准，也避免块太小丢上下文。
3. **向量化**：用 Embedding 模型把 chunk 转成向量，存入向量数据库，同时保存标题、来源、业务类型等 Metadata。
4. **检索**：用户问题进来后先做 query 改写或标准化，再做向量检索，必要时结合关键词检索和 Metadata 过滤。
5. **上下文拼装**：把 TopK 结果按相关性和业务优先级拼进 Prompt，要求模型基于上下文回答，不知道就明确说不知道。

如果面试官继续追问，我会补充：RAG 的关键不是“能搜到”，而是“搜得准、拼得稳、答得有依据”。所以要关注 chunk 粒度、召回策略、上下文长度和引用来源。

### 5. 面试官问：SSE 流式响应怎么做？为什么不用普通 HTTP？

普通 HTTP 要等模型完整生成后才能返回，用户会感觉等待时间长。LLM 应用通常生成时间不稳定，所以更适合流式返回。

我用 SSE 的原因是：

- 服务端可以持续推送 token 或阶段状态；
- 前端实现比 WebSocket 简单；
- 适合单向推送场景，比如模型生成、工具执行进度、最终结果；
- 可以明显改善首 token 延迟体验。

后端一般返回 `text/event-stream`，在生成过程中持续 yield 事件，例如：

- `message`：模型生成片段；
- `tool_start`：开始调用工具；
- `tool_result`：工具结果；
- `done`：任务完成；
- `error`：异常状态。

前端根据事件类型更新 UI，区分加载态、中间态、完成态和异常态。

### 6. 面试官问：LLM API 不稳定怎么办？

我会从四层处理：

1. **超时控制**：每次请求设置合理 timeout，避免一个模型请求卡死整个接口。
2. **重试策略**：对 429、5xx、网络抖动等可恢复错误做指数退避重试。
3. **降级兜底**：重试失败后返回可理解的业务提示，比如“当前模型服务繁忙，请稍后重试”，而不是接口直接崩。
4. **日志与观测**：记录请求耗时、错误类型、重试次数、模型返回，方便后续排查。

如果是生产系统，还可以加模型供应商 fallback、缓存常见问题答案、队列削峰和限流。

### 7. 面试官问：你怎么保证模型输出符合业务格式？

我会组合使用三种方式：

第一，Prompt 中明确输出格式，例如要求输出 JSON，字段包括 `intent`、`answer`、`actions`、`confidence`。

第二，用 Pydantic / JSON Schema 定义结构，模型返回后做解析和校验。校验失败时，可以让模型基于错误信息重新生成一次，或者走兜底逻辑。

第三，对关键业务字段做业务校验。比如日期是否合法、枚举值是否在允许范围内、工具参数是否缺失。格式正确不代表业务正确，所以不能只依赖 JSON parse。

### 8. 面试官问：如果 Agent 重复调用同一个工具怎么办？

我会在工程层做限制，而不是完全相信模型自己停下来。

常见做法：

- 设置最大执行步数；
- 对工具调用参数做指纹，检测最近几轮是否重复；
- 相同工具相同参数返回相同结果时，不继续重复调用；
- 重复达到阈值后注入提示，要求模型换思路；
- 严重时直接熔断，返回当前已知结果或失败原因。

本质上，Agent Loop 要有外层安全网：步数、时间、重复检测和异常兜底。

## 三、项目二：AI 实时语音对话智能客服

### 1. 面试官问：介绍一下实时语音客服项目

这是一个面向智能客服和实时交互场景的 AI 应用项目，目标是让用户通过语音或文本提出问题，系统能理解意图、检索业务知识，并实时生成回答。

整体链路是：语音输入先经过 ASR 转文字，然后进入意图识别和知识库检索，再由 LLM 基于上下文生成回复，最后通过流式方式返回前端，如果有语音输出需求，再接 TTS。

我重点关注的是对话链路的稳定性和体验：比如 Prompt 怎么区分意图识别、知识问答、业务引导和兜底回复；RAG 怎么召回准确资料；模型输出怎么结构化；前端怎么展示流式内容、加载态和异常态。

### 2. 面试官问：实时语音场景最大的挑战是什么？

最大的挑战是 **延迟、准确性和稳定性之间的平衡**。

语音客服链路比普通文本问答更长：ASR、意图识别、检索、LLM 生成、TTS，每一步都会增加延迟。为了让体验更自然，需要做几件事：

- 能流式的地方尽量流式，比如 LLM 输出和前端展示；
- Prompt 不要太长，RAG 上下文控制在必要范围；
- 常见问题可以缓存或走更轻量的意图路由；
- 对 ASR 识别错误要有容错，比如结合上下文改写 query；
- 对失败情况要有明确兜底，不要让用户一直等待。

### 3. 面试官问：客服 Prompt 怎么设计？

我会把客服 Prompt 分层设计：

1. **角色和边界**：说明这是业务客服，只回答业务范围内问题，不编造政策。
2. **任务类型**：区分知识问答、业务引导、信息收集、兜底回复。
3. **上下文约束**：要求优先基于检索资料回答；资料不足时说明无法确认。
4. **输出风格**：语气简洁、步骤清楚，适合用户直接理解。
5. **结构化字段**：必要时输出 intent、answer、need_human、confidence 等字段，方便系统后续处理。

客服场景最重要的是稳定和可信，不是让模型自由发挥。

### 4. 面试官问：答非所问怎么优化？

我会先判断问题出在哪一层：

- 如果是 ASR 识别错，就做 query 改写、上下文纠错或让用户确认；
- 如果是检索召回错，就调整 chunk、TopK、Metadata 过滤和关键词召回；
- 如果是 Prompt 约束不够，就强化“只能基于资料回答”和“资料不足要说明”；
- 如果是模型生成发散，就加结构化输出和后处理校验。

实际优化时不要直接怪模型，而是拆开看：输入、检索、上下文、Prompt、模型输出、后处理，每一层都有可能导致答非所问。

### 5. 面试官问：为什么实时客服里要做 RAG？

因为客服回答通常依赖业务规则、产品说明、活动政策、售后流程等动态知识。如果只靠模型参数记忆，答案可能过时或编造。

RAG 的价值是把外部知识注入到回答过程里，让模型基于最新业务资料生成答案。它还能降低幻觉，让回答可追溯。对客服系统来说，这比单纯让模型自由回答更安全。

## 四、项目三：共享雨伞业务系统与小程序架构优化

### 1. 面试官问：介绍一下你之前的业务系统项目

这个项目是共享雨伞业务的后台管理系统和支付宝 / 微信小程序，覆盖订单、设备、用户、活动配置和运营管理等核心场景。我主要负责前端业务模块开发、多端页面迭代、接口联调、通用组件封装、异常状态处理和线上问题排查。

这个项目的价值在于它是真实复杂业务系统，不是 demo。里面有订单状态流转、设备状态、支付 / 租借 / 归还流程、运营活动配置、多端兼容和后台管理效率等问题。我在这个项目里积累了比较扎实的业务系统开发、前后端协作和线上排查经验。

### 2. 面试官问：这个项目和 AI 应用岗位有什么关系？

它的直接技术栈不是 AI，但它提供了 AI 应用落地非常需要的业务工程经验。

AI 应用最终也要落到真实业务系统里，不只是一个聊天框。比如：

- 用户问题背后往往对应订单、设备、活动、规则等业务对象；
- Agent 调用工具时要理解业务状态和异常分支；
- 前端要展示加载、中间状态、错误状态和最终结果；
- 后端接口要考虑权限、幂等、超时和兜底；
- 线上出问题要能通过日志、接口和数据定位。

所以这段经历让我更理解 AI 应用不是单独的模型能力，而是模型能力和业务系统的结合。

### 3. 面试官问：你在前端项目里做过哪些优化？

主要有三类：

第一是 **组件和逻辑复用**。把后台管理系统里常见的表格、表单、搜索条件、状态展示、弹窗操作封装成通用组件，减少重复代码。

第二是 **联调效率优化**。建立 Mock 数据和异常场景，后端接口没完全 ready 时前端也能并行开发；同时提前覆盖空状态、错误状态、loading 状态。

第三是 **问题排查优化**。对于订单状态、设备状态这类问题，会结合接口返回、页面状态、用户操作路径和后端日志一起定位，而不是只看前端报错。

### 4. 面试官问：你怎么处理复杂业务状态？

我通常会先把状态枚举和状态流转梳理清楚。比如订单可能有待支付、租借中、已归还、异常结束等状态，设备也有在线、离线、可借、不可借、故障等状态。

前端不能只按“有没有数据”判断，而要根据业务状态明确展示不同操作入口和提示。复杂状态最好抽成统一的状态映射和工具函数，避免每个页面写一套 if else。

如果状态会影响后端操作，还要和后端约定好接口幂等、错误码和异常提示，避免用户重复点击或状态不同步造成问题。

## 五、高频综合追问答案

### 1. 你是前端转 AI 应用开发，优势是什么？

我的优势是能把 AI 能力产品化和工程化。很多 AI demo 只关注模型回答，但真实业务里还要考虑前端交互、接口设计、状态管理、错误处理、权限、日志和用户体验。我有前端业务系统经验，知道用户怎么使用系统，也在补 Python 后端、RAG 和 Agent，所以更适合做 AI 应用落地，而不是只做单点算法或单纯页面。

### 2. 你后端经验相比专业后端会不会弱？

我会承认传统后端深度还在补，但我不是从零开始。我熟悉 Web 系统开发、接口联调、业务状态和线上问题排查，现在重点补的是 Python / FastAPI、异步处理、Pydantic、数据库、Redis、超时重试、日志和部署。对于 AI 应用开发岗位，很多工作不是高并发底层系统，而是把 LLM、RAG、Agent 和业务接口稳定串起来，这正好是我当前重点准备的方向。

### 3. 你怎么理解 AI 应用开发？

AI 应用开发不是简单调用大模型 API，而是围绕业务目标构建一套可用、可控、可维护的系统。它至少包括：

- Prompt 和任务设计；
- LLM API 接入和异常处理；
- RAG 知识检索；
- Function Calling / Tool Calling；
- Agent 状态流转；
- 结构化输出校验；
- 前端流式交互；
- 日志、评估和迭代。

核心是把不稳定的模型能力放进稳定的工程边界里。

### 4. RAG 和微调有什么区别？什么时候用 RAG？

RAG 是在推理时检索外部知识，把相关内容放进上下文让模型回答；微调是改变模型参数，让模型学习某种能力或风格。

如果需求是业务知识经常变化、要求可追溯、资料可以更新，比如客服 FAQ、产品文档、内部制度，更适合 RAG。如果需求是固定风格、特定格式、领域表达习惯，且有大量高质量训练数据，可以考虑微调。

大部分企业知识问答优先选 RAG，因为成本低、更新快、可解释性更好。

### 5. 向量检索不准怎么办？

我会从几个方面排查：

1. **文档质量**：原始文档是否混乱、重复、过期。
2. **切分策略**：chunk 是否太大或太小，是否破坏语义边界。
3. **Query 质量**：用户问题是否口语化、缺少关键词，是否需要 query rewrite。
4. **召回策略**：是否只用向量检索，是否需要混合关键词检索。
5. **过滤条件**：是否可以用业务类型、时间、来源等 Metadata 缩小范围。
6. **排序优化**：是否需要 Rerank 提升 TopK 质量。

RAG 优化不是一个点，而是一条链路。

### 6. Agent 和普通工作流有什么区别？

普通工作流是开发者提前写死步骤，适合流程明确的任务。Agent 是模型根据当前状态决定下一步，比如是否检索、是否调用工具、调用哪个工具、是否继续追问。

但 Agent 不能完全放飞，工程上仍然要设置边界：工具 schema、权限、最大步数、超时、循环检测、输出校验和兜底策略。我的理解是：Agent 提供动态决策能力，工作流提供确定性边界，实际项目里通常是两者结合。

### 7. Function Calling / Tool Calling 的核心是什么？

核心是让模型从“只生成文本”变成“能选择工具并生成结构化参数”。但模型输出仍然不可信，所以后端必须做校验。

完整流程一般是：

1. 给模型工具列表和参数 schema；
2. 模型判断是否需要调用工具；
3. 模型生成工具名和参数；
4. 后端校验参数格式和业务合法性；
5. 后端执行工具；
6. 把工具结果回写给模型；
7. 模型基于结果生成最终回答。

重点是工具执行权在后端，不在模型。

### 8. 怎么减少大模型幻觉？

我会从输入、检索、约束和后处理四层控制：

- Prompt 明确要求不知道就说不知道；
- RAG 提供可靠上下文，并要求基于资料回答；
- 对关键字段做结构化输出和校验；
- 对高风险动作加人工确认或权限控制；
- 输出引用来源，方便用户判断可信度；
- 对常见问题建立评测集，持续评估回答质量。

不能完全消灭幻觉，但可以通过工程手段降低风险。

### 9. 你怎么做 AI 应用的评估？

我会分两类评估：

第一类是 **离线评估**，准备一批典型问题和标准答案，看召回是否准确、回答是否完整、是否引用正确资料、格式是否稳定。

第二类是 **线上观测**，记录用户问题、检索结果、模型回答、用户反馈、错误率、耗时、重试次数等指标。

对 RAG 来说可以评估 Recall@K、命中率、答案相关性；对 Agent 来说要关注工具调用成功率、循环次数、失败原因和任务完成率。

### 10. 你怎么处理流式输出中途失败？

前端不能假设流一定完整结束。后端 SSE 应该发送明确的 `error` 或 `done` 事件；前端根据事件状态决定是否展示“生成失败 / 重试”按钮。

后端如果模型调用中断，要捕获异常并推送错误事件，不能直接断开让前端不知道发生了什么。同时可以保留已生成内容，让用户决定是否重试。

### 11. 如果业务要求模型调用数据库，你会怎么设计？

我不会让模型直接写 SQL 或直接操作数据库。更安全的方式是把数据库能力封装成受控工具，例如 `query_order_status(order_id)`、`get_device_info(device_id)`。

模型只负责选择工具和生成参数，后端负责：

- 参数校验；
- 权限判断；
- SQL 执行；
- 超时限制；
- 结果脱敏；
- 日志记录。

如果确实要 Text-to-SQL，也要限制只读、白名单表、SQL 解析校验和最大返回行数。

### 12. AI 应用里 Redis 可以用在哪些地方？

常见用途有：

- 缓存高频 FAQ 或检索结果；
- 存储会话短期状态；
- 做任务队列或异步任务状态；
- 限流，比如按用户或 IP 控制请求频率；
- 分布式锁，避免重复处理同一个任务。

在 LLM 应用里，Redis 很适合放短生命周期、访问频繁、允许失效的数据。

### 13. PostgreSQL 和向量数据库怎么配合？

一般业务结构化数据放 PostgreSQL，比如用户、订单、任务、文档元信息；向量数据库存 chunk embedding 和检索索引。

也可以用 PostgreSQL + pgvector 做一体化方案，适合中小规模系统；如果数据量更大或检索性能要求更高，可以用专门的向量数据库。

关键是业务数据和向量 chunk 要有关联 ID，方便从检索结果追溯到原文档、来源和权限信息。

### 14. 你怎么做权限控制？

权限不能交给模型判断。模型只能提出“想做什么”，后端必须根据当前用户身份、角色、资源归属和工具权限做校验。

比如用户问订单信息，后端工具要确认这个用户是否有权限查看该订单；如果是写操作，还要增加确认步骤、审计日志和幂等控制。

AI 应用里的权限原则是：模型输出按不可信输入处理，所有副作用动作都必须经过服务端权限和业务校验。

### 15. 你项目里最大的收获是什么？

最大的收获是我对 AI 应用工程化有了更完整的理解。以前可能觉得接入模型、写 Prompt 就算 AI 应用；现在更清楚真实项目里要解决的是稳定性问题：检索是否准确、输出是否可控、工具是否安全、接口是否超时、前端是否能展示中间状态、异常是否能兜底。

所以我的关注点从“模型能不能回答”变成了“系统能不能稳定地帮助用户完成任务”。
