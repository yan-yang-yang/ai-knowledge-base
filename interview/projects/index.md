## 一、面试表达总原则

### 错误讲法

不要这样开场：

> 我之前主要做前端，最近学习了 FastAPI、RAG、Agent、LangGraph、SSE，所以做了一个 AI 业务运营系统。

问题：

- 一上来暴露「转型」痕迹；
- 面试官会觉得理论堆砌、缺少业务落地；
- RAG / Agent / SSE 像学习成果，不像项目方案；
- 没有解释为什么公共雨伞业务需要 AI。

### 正确讲法

应该这样开场：

> 我做的是一个面向公共雨伞借还业务的 AI 运营与客服辅助系统。这个业务的核心链路是用户扫码借伞、支付押金或免押、设备开锁、订单计费、归还识别、异常处理和客服售后。实际运营中会产生大量重复问题，比如用户借伞失败、归还后订单未结束、设备离线、雨伞丢失、活动规则咨询、网点补伞调度等。传统方式主要靠客服和运营人员查后台、翻规则、问技术，处理效率低，也容易漏掉上下文。所以我设计了一套基于 LLM + RAG + Agent Workflow 的智能运营系统，把业务知识检索、订单 / 设备查询、异常原因分析和处理建议整合到一个可控的 AI 流程里。

这段更有杀伤力，因为它先回答了：

1. 业务是什么；
2. 痛点是什么；
3. AI 解决什么；
4. 技术为什么合理。

## 二、整体项目自述

### 30 秒版本

我做的是一个面向公共雨伞借还业务的 AI 运营与客服辅助系统。公共雨伞业务链路比较长，涉及扫码借伞、设备开锁、订单计费、归还识别、押金 / 支付、异常售后和运营调度。实际工作里，客服和运营经常要处理借伞失败、归还未结束、设备离线、订单异常、活动规则咨询等问题。

所以我把 AI 能力落在这些业务问题上：通过 RAG 检索业务规则和处理 SOP，通过 Agent 调用订单、设备、网点等查询工具，再由 LLM 生成面向客服或运营人员的原因分析和处理建议。系统后端用 FastAPI，工作流用 LangGraph，结构化输出用 Pydantic，前端通过 SSE 实时展示模型分析过程和工具执行状态。

另外，公司内部也在基于 OpenClaw 探索数字人和研发提效能力。比如“小龙虾数字人”这个方向，本质不是只做一个会说话的形象，而是把 OpenClaw 的 Agent Loop、工具调用、多 Agent 协作、上下文记忆和多模态交互能力，迁移到公共雨伞客服、运营培训和代码开发提效场景里。对业务侧，它可以变成公共雨伞数字人客服 / 运营助手；对研发侧，它可以作为代码理解、接口联调、问题排查和文档生成的 AI 工程助手。

### 2 分钟版本

我这个项目不是单纯做一个聊天机器人，而是围绕公共雨伞借还业务做的 AI 运营与客服辅助系统。

公共雨伞业务看起来简单，其实链路比较长：用户扫码后要判断用户状态、押金或免押资格、设备在线状态、伞柜锁控状态；借出后要创建订单、开始计费；归还时要识别归还网点、更新订单状态；如果中间出现设备离线、锁控失败、用户归还后订单没结束、活动规则不一致，就会产生大量客服和运营问题。

传统处理方式是客服先看用户描述，再去后台查订单、查设备、查网点、翻活动规则或 SOP，复杂问题还要找技术排查。这种方式有三个问题：

1. **处理效率低**：一个问题可能要跨多个后台页面查询；
2. **规则容易不一致**：不同客服对押金、计费、异常退款、活动规则理解不一样；
3. **问题定位依赖经验**：比如借伞失败到底是设备离线、锁控异常、用户欠费，还是网点无伞，需要根据多源信息判断。

所以我设计了一套 AI 运营辅助流程：

1. 用户或客服输入问题，例如“用户说伞已经还了但订单还在计费”；
2. 系统先做意图识别，判断是订单异常、设备异常、规则咨询还是运营调度；
3. RAG 检索相关业务规则、售后 SOP、活动说明和异常处理文档；
4. Agent 根据意图选择工具，比如查询订单状态、设备状态、网点库存、用户历史记录；
5. LLM 结合检索资料和工具结果，生成原因分析、处理步骤、需要人工确认的风险点；
6. 通过 SSE 把“正在查询订单 / 正在检索规则 / 正在生成建议”等过程实时返回。

技术上，FastAPI 负责后端接口，Pydantic 约束请求参数、工具参数和模型结构化输出，LangGraph 管理多步骤状态流转，RAG 负责业务知识检索，SSE 负责流式展示。

这个项目的核心价值是：把 AI 从“泛泛问答”变成“能结合公共雨伞订单、设备、网点和规则的业务处理助手”。

## 三、项目一：公共雨伞 AI 运营与客服辅助系统

### 1. 面试官问：介绍一下这个项目

#### 回答

这是一个面向公共雨伞借还业务的 AI 运营与客服辅助系统，主要服务于客服、运营和内部处理人员。

公共雨伞业务里经常出现几类问题：

- 用户扫码借伞失败；
- 用户归还后订单没有结束，还在计费；
- 设备离线或锁控异常；
- 用户咨询押金、扣费、优惠活动规则；
- 某个网点无伞、满柜或需要补伞；
- 客服需要根据订单和设备状态判断是否退款、是否转人工或是否派运营处理。

这些问题的共同点是：它们不是单纯靠模型常识能回答的，必须结合业务规则、订单数据、设备状态和售后 SOP。

所以系统设计上，我把它分为四块：

1. **业务知识库**：存放借还规则、计费规则、押金规则、异常处理 SOP、活动说明等，通过 RAG 检索。
2. **业务工具调用**：把订单查询、设备查询、网点库存查询、用户状态查询封装成受控工具。
3. **Agent 工作流**：根据问题类型决定先检索规则，还是先查订单 / 设备，再生成处理建议。
4. **流式交互**：通过 SSE 展示处理过程，让客服知道系统正在做什么，而不是一直等待。

最终输出不是一句泛泛回答，而是结构化结果：问题类型、可能原因、证据来源、处理步骤、是否需要人工确认、建议回复话术。

### 2. 面试官问：这个项目的业务难点是什么？

我认为业务难点主要有四个。

第一，**借还链路长，异常原因多**。比如“借伞失败”可能是用户欠费、押金不足、设备离线、锁控失败、柜中无伞、活动资格不满足，也可能是第三方支付或小程序链路异常。如果 AI 只回答通用话术，没有结合订单和设备状态，就没有实际价值。

第二，**订单状态和设备状态需要联合判断**。比如用户说“我已经还伞了，但订单还在计费”，不能只看订单状态。还要看归还网点、设备上报记录、伞锁状态、最后一次心跳时间、是否存在网络延迟或设备离线。这个场景适合用 Agent 调用多个工具，再综合判断。

第三，**业务规则经常变化**。比如押金规则、优惠活动、免费时长、超时计费、异常退款政策可能随着运营策略调整。如果把规则写死在 Prompt 里，很容易过期，所以要用 RAG 从最新 SOP 和活动文档里检索。

第四，**客服回复要稳定、可解释**。客服不能拿一个不确定的模型回答直接回复用户，所以输出里要有证据来源、置信度、是否需要人工复核和推荐处理步骤。

### 3. 面试官问：项目亮点是什么？

我会讲三个亮点。

#### 亮点一：不是泛问答，而是业务状态驱动的 Agent

很多 AI 客服只能回答 FAQ，但这个项目会根据用户问题判断是否需要查业务状态。

比如用户问：

> 我明明还伞了，为什么还在扣费？

系统不会直接回答“请联系客服”，而是走一个 Agent 流程：

1. 识别意图为“归还异常 / 订单未结束”；
2. 调用订单查询工具，看订单是否仍在进行中；
3. 调用设备或网点工具，看归还点是否在线、是否有归还上报；
4. 检索 SOP，确认这种异常对应的处理规则；
5. 输出可能原因：设备离线导致归还事件延迟、归还未被识别、用户还到非指定点等；
6. 给客服处理建议：是否先安抚用户、是否需要冻结计费、是否转运营核查。

这个亮点体现的是：AI 和业务系统结合，不是只会聊天。

#### 亮点二：RAG 知识库按业务场景组织，而不是简单文档切片

我不会只把所有文档丢进向量库，而是按公共雨伞业务场景组织 Metadata：

- `borrow`：借伞规则、借伞失败处理；
- `return`：归还规则、订单未结束处理；
- `payment`：押金、扣费、退款、免押规则；
- `device`：设备离线、锁控失败、心跳异常；
- `station`：网点库存、满柜、缺伞、运营补伞；
- `activity`：优惠活动、免费时长、券规则；
- `after_sales`：客服 SOP、补偿规则、转人工条件。

检索时先做意图分类，再按业务类型过滤，避免“用户问归还异常，却召回活动规则”这种问题。

#### 亮点三：输出结构化，方便客服直接使用

最终回答不是一段散文，而是结构化输出，例如：

```json
{
  "intent": "return_order_not_closed",
  "risk_level": "medium",
  "possible_reasons": ["设备离线导致归还事件未及时上报", "订单状态未同步"],
  "evidence": ["订单当前状态：租借中", "归还网点最近心跳：10分钟前"],
  "suggested_actions": ["先告知用户正在核查", "查询设备归还记录", "必要时冻结异常计费"],
  "need_human_review": true,
  "reply_template": "您好，我们已看到您的订单仍在计费，会先为您核查归还记录，请您稍等。"
}
```

这样客服可以直接判断怎么处理，系统也能基于字段做后续流转。

### 4. 面试官问：RAG 在这个业务里怎么设计？

我的设计不是泛泛地“文档切片 + 向量检索”，而是围绕公共雨伞业务场景来做。

第一步是整理知识源：

- 借伞 / 还伞流程说明；
- 计费、押金、免押、退款规则；
- 设备异常处理 SOP；
- 订单异常处理 SOP；
- 活动规则、优惠券规则；
- 客服标准话术；
- 网点运营规则。

第二步是按业务语义切分。比如“归还后订单未结束”这类 SOP，要保留完整处理步骤，不能机械按固定长度切断，否则模型召回后会缺步骤。

第三步是给 chunk 加 Metadata，包括业务类型、适用场景、规则版本、是否客服可见、是否需要人工确认。

第四步是检索策略：

1. 先用 LLM 或规则做意图识别；
2. 根据意图加 Metadata 过滤；
3. 再做向量检索；
4. 必要时结合关键词，比如订单、押金、归还、离线、退款；
5. 最后把召回内容按 SOP 优先级拼进 Prompt。

这样可以提高召回准确率，也能避免模型拿错业务规则。

### 5. 面试官问：Agent 在这个业务里怎么工作？

我会用“订单未结束”举例。

用户输入：

> 我已经把伞还了，为什么订单还在计费？

Agent 流程大概是：

1. **意图识别节点**：判断为 `return_order_not_closed`。
2. **参数抽取节点**：抽取订单号、用户 ID、网点信息；如果缺少订单号，就返回需要补充的信息。
3. **订单查询工具**：查询订单当前状态、开始时间、计费状态。
4. **设备 / 网点查询工具**：查询归还网点设备在线状态、最近心跳、是否有归还记录。
5. **RAG 检索节点**：检索“归还后订单未结束”的 SOP。
6. **分析节点**：综合订单、设备、SOP，判断可能原因。
7. **结构化输出节点**：输出原因、证据、建议动作、客服话术。

这里 Agent 的价值是动态决策：有些问题只需要 RAG，有些必须查订单，有些还要查设备。不同意图走不同路径，比固定 Chain 更适合。

### 6. 面试官问：为什么用 LangGraph？

因为公共雨伞异常处理不是固定的一条链路，而是状态驱动的多分支流程。

比如：

- 活动规则咨询：只需要 RAG + 回复生成；
- 借伞失败：需要查用户状态、设备状态、网点库存；
- 归还未结束：需要查订单、设备上报、归还 SOP；
- 退款争议：需要查订单、支付状态、退款规则，并标记人工复核。

LangGraph 可以把这些流程拆成节点：意图识别、参数抽取、工具调用、RAG 检索、结果分析、人工复核判断、回复生成。每个节点输入输出明确，也方便做异常边、重试边和兜底边。

### 7. 面试官问：这个项目怎么处理模型输出不稳定？

公共雨伞业务里，模型输出不稳定会直接影响客服判断，所以我主要做三层控制。

第一层是 **结构化输出**。用 Pydantic 定义字段，比如 `intent`、`possible_reasons`、`evidence`、`suggested_actions`、`need_human_review`、`reply_template`。模型必须按结构返回。

第二层是 **业务校验**。比如风险等级只能是 low / medium / high；处理动作必须来自允许列表；如果涉及退款、冻结计费、订单关闭，必须标记人工复核，不能让模型直接决策。

第三层是 **兜底策略**。如果模型输出解析失败、证据不足或工具调用失败，就返回“需要人工核查”，同时给出已查到的信息，而不是编造答案。

### 8. 面试官问：你怎么避免 AI 乱操作订单？

核心原则是：模型不能直接操作订单，所有副作用动作必须经过后端权限和人工确认。

系统里工具分两类：

- **只读工具**：查询订单、设备、网点、用户状态，AI 可以在权限范围内调用；
- **写操作工具**：退款、关闭订单、冻结计费、派单处理，这类只能生成建议，不能自动执行，或者必须经过人工确认。

即使后续支持自动化，也要有：

- 用户身份和角色权限；
- 工具白名单；
- 参数校验；
- 幂等控制；
- 操作审计日志；
- 高风险动作二次确认。

这能体现我对 AI 工程安全边界的理解。

### 9. 面试官问：SSE 在这个项目里的价值是什么？

客服处理异常时，如果接口要等 10 秒才返回完整结果，体验会很差，也不知道系统卡在哪里。SSE 可以把处理过程分阶段推出来：

- `intent_detected`：已识别为归还异常；
- `tool_start`：正在查询订单；
- `tool_result`：订单状态为租借中；
- `rag_search`：正在检索归还异常 SOP；
- `analysis`：正在生成处理建议；
- `done`：输出最终建议。

这样客服会感觉系统在持续工作，也方便定位是哪一步慢，比如订单接口慢、设备接口慢还是模型生成慢。

### 10. 面试官问：你怎么评估这个项目效果？

我会从业务和 AI 两个维度评估。

业务维度：

- 客服平均处理时长是否下降；
- 常见问题一次解决率是否提升；
- 订单异常定位时间是否缩短；
- 是否减少客服反复询问技术或运营；
- 高风险问题是否能正确转人工。

AI 维度：

- 意图识别准确率；
- RAG 召回命中率；
- 工具调用成功率；
- 结构化输出解析成功率；
- 幻觉率或无依据回答比例；
- Agent 平均执行步数和失败原因。

这类项目不能只看模型回答好不好，还要看它有没有真正降低业务处理成本。

## 四、项目二：公共雨伞实时语音智能客服

### 1. 面试官问：介绍一下实时语音客服项目

这个项目是把公共雨伞售后咨询做成实时语音 / 文本客服助手。用户可能会问：

- 怎么借伞、怎么还伞；
- 为什么借不了；
- 为什么还了还在扣费；
- 押金什么时候退；
- 优惠券为什么不能用；
- 附近哪里有可借雨伞。

整体链路是：ASR 把语音转成文本，系统做意图识别，然后根据问题类型走 RAG 或工具调用，最后用 LLM 生成适合客服场景的回复。如果是语音场景，还可以接 TTS 输出。

我重点解决的是实时体验和业务准确性：既要回答快，又不能脱离公共雨伞的规则和订单状态乱回答。

### 2. 面试官问：语音客服难点在哪里？

有三个难点。

第一，**语音识别可能不准确**。比如用户说“还伞”，ASR 可能识别成“换伞”；用户口语里也可能没有订单号，只说“我刚才还了怎么还扣钱”。所以系统要结合上下文做 query 改写或追问。

第二，**实时链路长**。ASR、意图识别、RAG、工具查询、LLM、TTS 每一步都有延迟。要通过流式返回、短 Prompt、缓存高频规则、轻量意图分类来降低等待感。

第三，**客服回答必须受业务约束**。比如退款、押金、扣费不能随便承诺，必须基于订单状态和公司规则。资料不足时，要引导用户提供订单号或转人工。

### 3. 面试官问：客服 Prompt 怎么设计？

我会按公共雨伞客服场景设计 Prompt，而不是通用客服 Prompt。

核心约束包括：

1. 只回答公共雨伞借还、订单、押金、计费、设备和活动相关问题；
2. 涉及退款、免单、冻结计费等高风险动作，只能给建议或转人工，不能直接承诺；
3. 如果缺订单号、用户 ID、网点信息，要先追问必要信息；
4. 优先基于 RAG 检索到的 SOP 和工具查询结果回答；
5. 输出要简洁，适合客服直接复制给用户。

### 4. 面试官问：如何处理“用户已还伞但仍扣费”的语音问题？

我会把它识别为归还异常类问题。

如果用户没提供订单号，先追问订单信息；如果有订单号，就查询订单状态、归还记录、设备心跳和网点信息。然后检索对应 SOP，判断是设备上报延迟、归还未识别、还错网点、订单状态同步失败，还是需要人工核查。

回复时不会直接说“一定给你退款”，而是会说：

> 我先帮您核查订单和归还记录。如果确认是设备或系统同步导致的异常计费，会按规则为您处理；目前需要您提供订单号或手机号后四位。

这样既安抚用户，又避免模型过度承诺。

## 五、项目三：基于 OpenClaw 的公共雨伞数字人与研发提效系统

### 1. 面试官问：介绍一下这个 OpenClaw 数字人项目

这个项目可以这样讲：公司内部在做基于 OpenClaw 的数字人能力探索，最开始有“小龙虾数字人”这类 Demo 或样板项目，用来验证数字人形象、语音交互、Agent 工具调用和多轮对话能力。但我在项目表达里不会把它讲成餐饮业务，而是讲它背后的技术能力如何迁移到公共雨伞业务和研发流程里。

我会把它定位为 **基于 OpenClaw 的公共雨伞数字人客服与研发提效系统**。

业务侧，它可以作为公共雨伞数字人客服 / 运营助手，接入借还规则、押金计费、设备异常、订单售后、网点运营 SOP。用户或客服通过语音 / 文本提问，数字人负责理解意图、检索业务规则、调用订单和设备工具，然后用更自然的方式给出解释、引导和处理建议。

研发侧，它可以作为内部代码开发提效助手，接入项目仓库、接口文档、错误日志和业务知识。研发人员可以让它帮忙理解公共雨伞订单状态流转、定位设备接口异常、生成接口 Mock、整理联调文档、解释报错日志、辅助编写测试用例。

所以这个项目的重点不是“数字人皮肤”，而是 OpenClaw 作为 Agent Runtime，如何把模型、工具、上下文、记忆、多模态交互和业务系统连接起来。

### 2. 面试官问：为什么公共雨伞业务适合做数字人？

公共雨伞业务有几个特点，很适合数字人或语音客服形态。

第一，用户问题高频且口语化。比如“我还了伞怎么还扣钱”“为什么扫了码打不开”“押金什么时候退”“附近哪里能借伞”。这些问题天然适合语音 / 对话式入口。

第二，问题背后需要业务状态判断。数字人不能只回答 FAQ，还要能结合订单、设备、网点、用户状态判断原因。这正好适合 OpenClaw 的工具调用能力。

第三，客服和运营也需要培训与辅助。新客服不一定熟悉所有借还规则、退款边界和设备异常 SOP，数字人可以作为内部业务教练，解释规则、演示处理流程、生成客服话术。

第四，公共雨伞属于线下设备业务。用户遇到问题时往往比较着急，如果数字人能实时解释处理进度，比如“正在查询订单状态”“正在检查设备心跳”“需要人工核查退款”，体验会比静态 FAQ 更好。

### 3. 面试官问：这个项目的技术架构怎么设计？

我会分成六层讲。

第一层是 **交互入口**。支持 Web / 小程序客服入口，也可以接数字人形象、ASR 语音识别和 TTS 语音播报。

第二层是 **OpenClaw Agent Runtime**。OpenClaw 负责 Agent Loop、工具调用、事件流、上下文管理和权限策略。它不是简单把问题发给模型，而是让模型可以在受控边界内调用工具完成任务。

第三层是 **业务工具层**。把公共雨伞业务系统封装成工具，比如查询订单状态、查询设备心跳、查询网点库存、查询押金支付状态、检索售后 SOP。

第四层是 **RAG 知识库**。存放借还规则、计费规则、押金退款规则、设备异常 SOP、活动规则、客服话术和研发文档。

第五层是 **数字人表达层**。把 Agent 的结构化结果转成适合用户理解的自然语言，必要时驱动数字人口型、语音播报和状态提示。

第六层是 **研发提效插件层**。接入代码仓库、接口文档、日志和测试命令，用于代码问答、接口联调、错误排查、Mock 生成和文档沉淀。

### 4. 面试官问：OpenClaw 在这里解决什么问题？

OpenClaw 解决的是“让数字人从聊天界面变成能行动的 Agent”。

如果只是普通 Chat API，数字人只能根据 Prompt 回答问题。但公共雨伞业务需要查实时订单、查设备状态、查网点库存、检索 SOP，还要控制权限和风险。OpenClaw 的价值在于：

- **Agent Loop**：支持多轮观察、思考、调用工具、生成结果；
- **Tool System**：把订单、设备、网点、知识库、代码仓库封装成工具；
- **事件流**：把工具执行进度实时推给前端或数字人；
- **Context Engineering**：控制长对话和工具结果，避免上下文爆炸；
- **Memory / 知识沉淀**：沉淀用户偏好、客服 SOP、研发经验；
- **权限策略**：限制高风险工具，比如退款、关闭订单、生产写操作。

所以 OpenClaw 更像数字人的运行底座，而不是一个普通聊天 SDK。

### 5. 面试官问：业务侧难点是什么？

业务侧最大难点是 **数字人回答必须结合真实业务状态，而不能只讲通用话术**。

比如用户问：

> 我刚刚已经还伞了，为什么订单还在扣费？

数字人需要做的不是直接安抚，而是：

1. 识别为归还异常；
2. 追问或抽取订单号；
3. 调用订单工具查询订单是否仍在计费；
4. 调用设备工具查询归还设备是否在线、是否上报归还事件；
5. 检索归还异常 SOP；
6. 判断是否可能是设备离线、归还事件延迟、还错网点或订单同步异常；
7. 输出用户话术和客服处理建议；
8. 涉及退款或冻结计费时转人工确认。

这要求数字人既要自然交互，又要严格受业务规则约束。

### 6. 面试官问：研发提效侧怎么落地？

研发提效侧，我会围绕公共雨伞业务系统的真实研发流程讲。

这个业务里有后台管理系统、小程序、订单接口、设备接口、支付押金、活动配置、售后处理等模块。研发过程中经常遇到几类低效问题：

- 新人理解订单 / 设备状态流转慢；
- 前后端联调时接口字段和状态码反复确认；
- 设备异常或线上报错需要跨日志、接口、业务状态排查；
- 活动配置、售后 SOP、接口文档分散，查找成本高；
- 重复写 Mock、测试用例、联调说明和排查记录。

OpenClaw 可以作为研发 Agent 做这些事：

- 基于代码仓库回答“订单状态在哪里流转”；
- 根据接口文档生成 Mock 数据和 TypeScript 类型；
- 根据报错日志定位可能的接口或状态分支；
- 根据需求说明生成开发任务拆解和测试点；
- 把线上问题排查过程沉淀成知识库；
- 在代码修改前检索相关模块，减少漏改。

我会强调：研发提效不是让 AI 直接替代开发，而是让 AI 降低理解成本、检索成本、联调成本和文档成本。

### 7. 面试官问：这个项目亮点是什么？

我会讲四个亮点。

第一，**数字人和业务工具打通**。数字人不是只播报话术，而是能通过 OpenClaw 调用订单、设备、网点和知识库工具，回答基于实时业务状态。

第二，**同一套 Agent 能力同时服务业务和研发**。业务侧用于客服、运营、培训；研发侧用于代码理解、联调排查、文档生成。底层都是 OpenClaw 的工具调用、上下文管理和事件流能力。

第三，**可解释的处理过程**。通过事件流把“识别意图、查询订单、检查设备、检索 SOP、生成建议”展示出来，用户和客服都知道系统为什么这么回答。

第四，**高风险动作有边界**。涉及退款、关闭订单、冻结计费、生产数据修改，数字人只能给建议或发起人工确认，不能直接越权执行。

### 8. 面试官问：这个项目难点是什么？

难点主要有五个。

第一，**多模态延迟控制**。语音输入、模型推理、工具调用、语音播报、数字人渲染都会增加延迟，需要流式返回和分阶段反馈。

第二，**上下文管理**。客服对话、业务工具结果、RAG 文档、代码片段都可能很长，需要截断、摘要和按需注入，避免模型上下文失控。

第三，**工具调用可靠性**。订单、设备、网点工具可能超时或失败，Agent 要能重试、降级，并告诉用户当前能确认什么、不能确认什么。

第四，**业务规则和实时数据融合**。RAG 解决规则问题，工具解决实时状态问题，最终回答必须把两者结合，而不是只依赖其中一个。

第五，**安全边界**。数字人形态容易让用户误以为它能直接承诺退款或处理订单，所以系统必须限制高风险动作，输出中明确人工复核条件。

### 9. 面试官问：如果问到“小龙虾数字人”和公共雨伞有什么关系，怎么回答？

可以这样答：

> 小龙虾数字人更像公司内部验证数字人交互和 OpenClaw Agent 能力的样板项目，它验证的是数字人形象、语音对话、工具调用和多轮任务处理这些通用能力。我在面试里重点讲的是，这套能力如何迁移到公共雨伞业务：把数字人从餐饮展示 Demo，变成能处理借伞失败、归还异常、押金退款、设备故障和运营培训的业务数字人。同时，OpenClaw 的 Agent 能力也能用于研发提效，比如代码理解、接口联调、日志排查和知识库沉淀。

这段回答的重点是：不要硬说小龙虾就是公共雨伞业务，而是说它是技术验证样板，公共雨伞是业务落地场景。

### 10. 面试官问：这个项目怎么评估效果？

业务侧看：

- 数字人问题理解准确率；
- 借还规则问答命中率；

- 转人工是否更准确；

### 11. 项目一句话总结

一句话可以这样说：

## 六、项目四：公共雨伞业务系统与 AI 落地基础

### 1. 面试官问：你之前的业务系统经验怎么支撑 AI 项目？

我会把这段经历讲成 AI 项目的业务基础，而不是单独讲前端。

公共雨伞业务系统覆盖扫码借伞、订单计费、归还识别、设备管理、网点运营、活动配置和客服售后。做 AI 项目前，我已经理解这套业务链路，所以我知道 AI 不能只做通用问答，而要和订单、设备、用户、网点、活动规则结合。

比如：

- 用户借不了伞，要查用户资格、设备状态、网点库存；
- 用户还伞后仍计费，要查订单状态、设备上报和归还规则；
- 用户问押金退款，要查支付状态、押金规则和退款 SOP；
- 运营问哪个网点缺伞，要结合网点库存和历史借还数据。

这段业务经验让我在设计 AI 应用时，能自然地把问题拆成“业务意图 + 数据查询 + 规则检索 + 处理建议”，而不是停留在理论层面。

### 2. 面试官问：公共雨伞业务里有哪些适合 AI 的场景？

我会分四类。

第一类是 **客服问答**：借还规则、押金、计费、活动、退款等问题，可以用 RAG + 话术生成。

第二类是 **异常诊断**：借伞失败、归还未结束、设备离线、订单异常，可以用 Agent 调用订单 / 设备 / 网点工具做综合判断。

第三类是 **运营辅助**：网点缺伞、满柜、设备异常聚合、补伞优先级建议，可以结合数据分析和 LLM 生成运营建议。

第四类是 **知识库维护**：把客服 SOP、活动规则、设备异常文档整理成结构化知识，辅助新客服培训和内部查询。

### 3. 面试官问：你怎么把业务系统抽象成 AI 工具？

我会按业务对象封装工具，而不是直接暴露数据库。

例如：

- `get_order_status(order_id)`：查询订单状态、计费状态、借还时间；
- `get_device_status(device_id)`：查询设备在线状态、心跳时间、锁控状态；
- `get_station_inventory(station_id)`：查询网点可借 / 可还数量；
- `get_user_risk_status(user_id)`：查询用户是否欠费、是否有未完成订单；
- `search_policy(intent)`：检索对应业务 SOP。

模型只负责选择工具和生成参数，真正的权限校验、SQL 查询、结果脱敏和异常处理都在后端完成。

### 4. 面试官问：这个业务里最能体现工程能力的点是什么？

我会说是“多源状态联合判断 + 可控自动化”。

公共雨伞异常往往不是单一数据能判断的，比如订单未结束可能涉及订单状态、设备上报、网点归还记录、用户描述和 SOP 规则。AI 的价值是把这些信息汇总起来，但工程上必须保证：

- 查询工具可靠；
- 参数和权限可校验；
- 模型不能越权执行；
- 结果要有证据；
- 高风险处理要人工复核；
- 失败时要能兜底。

这比单纯做一个 ChatBot 更接近真实 AI 应用开发。

## 七、高频综合追问答案

### 1. 你怎么理解 AI 应用开发？

AI 应用开发不是简单接大模型 API，而是把模型能力嵌入具体业务流程，形成可用、可控、可评估的系统。

以公共雨伞业务为例，AI 应用不是让模型回答“怎么借伞”这么简单，而是要能理解“借伞失败”“归还未结束”“押金退款”“设备离线”这些业务意图，然后结合订单、设备、网点和 SOP 给出处理建议。

所以 AI 应用开发包含：业务建模、RAG 知识库、工具调用、Agent 流程、结构化输出、权限控制、流式交互、日志评估和持续迭代。

### 2. RAG 和业务怎么结合？

RAG 一定要和业务场景结合，不能只是把文档切片丢进向量库。

在公共雨伞业务里，我会按借伞、归还、支付押金、设备异常、网点运营、活动规则、售后 SOP 来组织知识库，并给每个 chunk 加 Metadata。用户问题进来后先识别意图，再用业务类型过滤，最后做向量召回。

这样可以避免召回错场景的规则，比如用户问归还异常，却召回到优惠券活动说明。

### 3. 向量检索不准怎么办？

我会从业务链路排查：

1. 是否先做了意图分类；
2. chunk 是否按完整 SOP 切分，而不是切断步骤；
3. Metadata 是否覆盖 borrow / return / payment / device / station 等场景；
4. 是否需要关键词补充，比如“扣费”“押金”“还伞”“离线”；
5. 是否需要 Rerank；
6. 是否存在过期规则，需要版本字段过滤。

核心是让检索服务业务判断，而不是只追求语义相似。

### 4. Agent 和普通工作流有什么区别？

普通工作流适合固定步骤；Agent 适合根据问题动态选择下一步。

公共雨伞里，不同问题路径不一样：

- 规则咨询：RAG 就够；
- 借伞失败：要查用户、设备、网点；
- 归还未结束：要查订单、设备上报、归还 SOP；
- 退款争议：要查支付状态和退款规则，并进入人工复核。

所以我会用 LangGraph 把 Agent 做成可控的状态机，而不是让模型无限自由调用工具。

### 5. Function Calling / Tool Calling 的核心是什么？

核心是让模型能选择业务工具，但执行权必须在后端。

在公共雨伞项目里，模型可以选择 `get_order_status`、`get_device_status`、`search_policy` 这些工具，但工具参数要经过 Pydantic 校验，用户权限要在后端判断，返回结果要脱敏，高风险动作不能直接执行。

一句话：模型负责决策建议，后端负责安全执行。

### 6. 怎么减少大模型幻觉？

我会结合业务证据控制幻觉：

- Prompt 要求基于订单、设备和 SOP 回答；
- RAG 召回规则必须带来源；
- 工具结果作为事实输入；
- 输出里要包含 evidence 字段；
- 资料不足时必须说无法确认；
- 涉及退款、冻结计费、关闭订单必须转人工确认。

这样模型即使生成自然语言，也不能脱离业务证据乱说。

### 7. 你怎么做权限控制？

权限不能交给模型判断。

公共雨伞业务里，查询订单、查看用户信息、退款、关闭订单、冻结计费都是敏感能力。我的设计是：

- 模型只生成工具调用意图；
- 后端根据当前客服或运营角色判断权限；
- 只读工具和写工具分级；
- 写操作默认人工确认；
- 所有操作记录审计日志；
- 工具返回结果做脱敏。

### 8. 你怎么处理 LLM API 不稳定？

我会做超时、重试、降级和观测。

比如客服正在处理用户订单异常，如果模型接口卡死，不能让客服一直等。所以每次 LLM 请求要设置 timeout；对网络抖动和 5xx 做指数退避；失败时返回兜底建议，例如“当前 AI 分析失败，已展示订单和设备查询结果，请转人工处理”；同时记录耗时、错误类型和重试次数。

### 9. SSE 在业务里解决什么问题？

SSE 解决的是客服等待过程不可见的问题。

公共雨伞异常诊断可能要查订单、查设备、检索 SOP、生成建议。如果一次性返回，客服不知道系统是否卡住。SSE 可以实时推送：正在识别问题、正在查订单、正在查设备、正在检索规则、正在生成建议。

它不只是技术优化，而是提升客服处理体验和可观测性。

### 10. 你怎么评估 AI 项目？

不能只评估模型回答，要评估业务效果。

公共雨伞项目里，我会看：

- 借还异常平均处理时间；
- 客服一次解决率；
- SOP 召回命中率；
- 工具调用成功率；
- 结构化输出解析成功率；
- 高风险问题转人工准确率；
- 用户投诉或重复咨询是否下降。

AI 指标要最终回到业务效率和处理质量。

可以这样回答：

### 12. 如果面试官问：你最大的项目亮点是什么？

我的最大亮点是把公共雨伞业务异常处理抽象成了“意图识别 + 多源工具查询 + RAG 规则检索 + 结构化处理建议”的 AI 工作流。

它解决的不是泛泛聊天，而是客服和运营真实会遇到的问题，比如借伞失败、归还未结束、设备离线和押金退款。技术上用 Agent 处理动态路径，用 RAG 保证规则准确，用 Pydantic 保证输出稳定，用权限和人工复核保证安全边界。

### 13. 如果面试官问：OpenClaw 数字人项目和公共雨伞业务怎么结合？

我会说，小龙虾数字人验证的是 OpenClaw 的数字人交互和 Agent 能力，公共雨伞是更具体的业务落地场景。

迁移到公共雨伞后，数字人不只是展示形象，而是可以处理借伞失败、归还异常、押金退款、设备离线、网点查询和客服培训。它背后通过 OpenClaw 调用订单、设备、网点和知识库工具，结合 RAG 检索 SOP，再生成可解释的处理建议。

所以这个项目的价值不是“做了一个数字人”，而是把数字人变成公共雨伞业务里的 Agent 入口。

### 14. 如果面试官问：OpenClaw 对开发代码提效体现在哪里？

我会结合公司公共雨伞业务系统讲。

研发提效主要体现在四点：

1. **代码理解**：快速解释订单、设备、支付、活动等模块的代码结构和状态流转；
2. **接口联调**：根据接口文档生成 Mock、类型定义、测试请求和异常场景；
3. **问题排查**：结合报错日志、接口返回和业务状态，定位可能的问题分支；
4. **知识沉淀**：把排查过程、接口约定、业务规则沉淀到知识库，减少重复沟通。

OpenClaw 的优势是它不只是聊天，而是可以读代码、查文档、跑命令、调用工具，并把这些动作放在一个可控的 Agent 流程里。

> 我做的不是脱离业务的 AI Demo，而是把公共雨伞借还业务中的异常处理和客服运营流程，抽象成可检索、可调用、可解释、可管控的 AI 工作流。
